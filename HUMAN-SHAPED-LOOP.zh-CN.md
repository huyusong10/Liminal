# Human-Shaped Loop：为什么长期 Agent 任务需要人的判断结构

**简体中文** | [English](./HUMAN-SHAPED-LOOP.md)

本文阐述 Loopora 背后的工程思考与协作理念。想安装、运行和使用 Loopora，请参见 [README](./README.zh-CN.md)。

Loopora 的出发点很朴素：偷懒。

更准确地说，是不想一直守在电脑前，等 Agent 跑完一轮，再指出哪里不对、催它去改。

这里想省掉的并非判断本身，而是把同一类判断重复做十几遍。

简单讲，Loopora 的目的不是帮你写更长的 PRD，也不是让 Agent 再多做几轮随意的自我反省。它在 Agent 之外构建了一套长期任务的运行结构：先明确任务应如何被判断，再要求后续每一轮都用证据回到这套判断上。

## 1. 一个看起来很适合 Agent 的任务

想象一家 B2B SaaS 公司，客服团队每天处理大量退款工单。

团队决定做一个退款自助流程：客户管理员打开账单页，看到某笔订单符合退款条件，即可提交退款请求，并得到明确结果。若订单存在风险，流程就转交客服处理。

这看起来很适合交给 Coding Agent：

- 有产品界面要做。
- 有业务规则要编码。
- 有测试要补。
- 有边界情况要发现。
- 工作量足够大，一轮很可能不够。

于是用户提出需求：

> 做一个退款自助流程：客户管理员可以在账单页申请符合条件的订单退款；有风险的订单转交客服。做得安全一点，补上测试，迭代到可以作为正式功能交付为止。

第一轮结果看起来不错：页面有了，表单有了，状态提示有了，退款资格判断用了几组模拟规则，主路径测试也都通过了。

Agent 回复：

> 已经完成了！我达成了目标！

第一眼看上去，这个结果已经完整。页面在，按钮在，流程也能走通。如果只是 demo，故事或许到此为止。

但若作为正式产品上线，真正的问题才刚刚开始：

- 它并未证明只有授权客户管理员可以发起退款。
- 退款资格判断只是几组模拟规则，并非可靠的业务路径。
- 主路径测试虽已通过，但并未覆盖部分退款、争议订单、拒付、超出退款窗口、财务关账等情形。
- 它未说明支付服务商执行失败时，系统记录、账务状态和客服接管应如何处理。
- 它也未证明审计日志足以让客服、财务或合规在事后还原全过程。

此时人类 reviewer 面对的不是抽象问题，而是一个具体的上线抉择：这个功能现在能否交付？

答案是还不能。

于是 reviewer 表示：

> 这还不能交付。先证明权限、退款资格、支付失败和审计链路，再考虑继续打磨 UI。

第二轮结果也可能看起来更像正式产品。它补了更多页面状态、更多确认文案、更多模拟数据，也在总结里提到"权限""资格""审计"这些词。

问题恰恰出在这里：人类的提醒没错，Agent 也不是完全没听进去。但在普通对话中，这个判断始终只是一段静态文本。它尚未分解为必须引用的证据、不可放行的阻断条件、下一轮的优先级，以及阻止收尾的终止条件。

因此 Agent 可能在局部回应了提醒，但很快又退回更容易完成的工作：把权限写进文案，把资格判断继续留在模拟规则里，为 happy path 补更多测试，然后用更自信的语气称"安全性已加强"。它做了不少事，核心风险却几乎没有减少。危险并未消失，只是被更像产品的界面和更圆满的总结叙事掩盖了。

如果继续沿这个方向多跑几轮，结果可能不会突然失控，而是每一轮都"合理地"一点点跑偏：第一轮把"页面能提交"当成完成，第二轮把"提到了权限"当成证明，第三轮围绕容易通过的路径补测试，第四轮开始整理 README 和最终总结。结果越来越像正式功能，但退款安全仍然没有被证明。

Agent 并非不够勤奋，轮次也不算少，人类的提醒本身也没错。真正缺的是：正确的判断缺少一种能在后续轮次中持续生效的运行形态。

人类每次介入，不仅是在补充需求，更是在向系统施加控制：拒绝什么、信任什么、拦截什么、下一步改什么、何时收尾。长期任务真正消耗人的，是这些控制需要一次又一次手动施加。

## 2. 为什么 PRD、用例和普通循环还不够？

任务开始前就把所有问题考虑清楚，这有可能吗？不少 AI harness 和 prompt 工作流做的就是这件事。

### PRD 写得再细，也替代不了执行中的门禁

如果一开始问得更充分、拆得更清楚、写一份好 PRD，再让 Agent 去 plan / execute / self-review，结果通常确实会好很多。

前期澄清充分，能降低第一轮跑偏的概率。PRD 可以阐明目标、约束和范围。checklist 可提醒 Agent 不要遗漏常见风险。测试与 benchmark 能将一部分判断转化为硬性反馈。多 Agent review 也能增加怀疑视角。

这些方法都值得采用。但它们主要改善的是"开始前的信息质量"或"某类硬验证"。它们降低了初始误差，并不等于建立了执行中的纠偏机制。

在真实工程团队里，设计文档写得再细，也不会有人因此取消 code review 和 CI 门禁。文档可以阐明意图，却无法在执行过程中自动拦住偏离。Agent 任务中的 PRD 和 prompt 面临同样的局限：它们能描述目标和约束，却无法保证执行者不会在每个岔路口选错方向，更无法在跑偏之后自动暴露偏差、补齐证据、承认尚未完成。

退款任务进入多轮执行后，每一轮都会产生新的事实：

- 它实际改了哪些代码和流程？
- 它绕过了哪些难点？
- 它新增的测试是在证明核心风险，还是只证明更容易通过的路径？
- 它的总结是否把"没有证明"写成了"已经完成"？

这些事实只有在执行中才会出现，因此每一轮都必须回答以下控制问题：

- 这一轮证明的是退款业务安全，还是只证明页面可提交？
- 支付失败、重复退款、越权退款等问题是否必须拦截？
- 哪些缺口可以作为显式残余风险，哪些不能带着走？
- 下一轮应该扩展、补证据、收窄范围、修复根因，还是停止？

**静态输入与执行控制的差别如下：**

| PRD / prompt | Human-shaped Loop |
| --- | --- |
| 描述任务开始前已知的目标与约束 | 将约束转化为执行中持续生效的控制结构 |
| 提醒 Agent 应注意哪些事项 | 要求每轮以证据回应这些事项 |
| 改善第一轮质量 | 控制多轮之间的误差传播 |
| 可被选择性引用或局部满足 | 记录缺口、阻断问题与残余风险 |
| 主要回答"要做什么" | 持续回答"是否已证明、是否该转向、能否收尾" |

### 说一次远远不够

这不是 AI 独有的问题。一份 PRD 在工程里也不是终局。

若一个工程团队要上线退款流程，他们当然会先澄清需求、撰写设计、列出风险、补充用例。但即便是最强的团队，也不会指望产品经理、工程师和 reviewer 在第一天就穷尽所有问题。许多事实注定只有在实现过程中才会暴露：某个权限路径与真实组织模型不匹配，某个支付失败状态与账务系统冲突，某个测试看似覆盖了边界，实则绕开了最危险的分支。

因此成熟的工程流程中包含设计评审、代码评审、测试、灰度发布、上线门禁、监控和复盘。这些环节之所以存在，不是因为前期澄清不重要，而是"一开始就能说清一切"本就是伪命题。工程判断必须随着新证据不断回调到任务本身。

Loopora 的很多理念正是从这类工程经验里长出来的：需求不是一次性写完就生效的文档，而是一组会在执行过程中反复应用的判断。长期 Agent 任务也一样。它不只需要更好的开场白，还需要一种机制，让后续每一轮都继续受同一组工程判断的约束。

### 固定用例有用，但它只守护已知边界

退款任务里，有些东西当然应该变成固定用例：

- 只有授权管理员能退款。
- 超出退款窗口不能退款。
- 重复退款必须被拒绝。
- 支付失败必须留下记录并进入接管路径。

这些用例越硬越好。Loopora 不会替代它们。

但固定用例主要守护已知、稳定、可枚举的边界。这跟工程里的单元测试很像：单元测试覆盖了函数输入输出的已知边界，但无法回答集成后才会暴露的未知问题。长期任务中还有更上层的问题：

- 用例失败了，下一轮应该先修哪里？
- 用例没覆盖，但证据明显不足，能不能收尾？
- Agent 新增的测试是在证明核心风险，还是绕过难点？
- 新出现的缺口是阻断风险，还是可见的残余风险？
- 当固定用例不完整时，系统如何要求 Agent 暴露"未证明"？

> 固定用例守护已知边界；Human-shaped Loop 管理长期任务中不断出现的判断、证据与转向。

### 没有门禁的循环，跑再多轮也拦不住偏差

当前让 Agent 多轮运行的方法已经不少：重复调用同一个 Agent、让模型自我审查、给它 checklist 再继续、跑测试再修复。

这些方法在外部反馈明确时很管用。benchmark、contract test、schema、lint、type check 和 proof harness 都能给出确定反馈。当判断已经被这些工具表达清楚，简单循环就够了。

但退款流程的判断很难压缩成单一分数。测试通过了不代表业务流程安全，界面精美也可能掩盖一个事实：退款权限、审计、支付服务商失败和客服接管均未被证明。

这跟一个没有门禁规则的 CI 流水线很像：代码可以反复提交、反复跑测试，但只要缺少强制性的准入检查，有问题的代码依然可以合入主干。Agent 在多轮循环中虽然多次行动，却始终无法稳定回答几个关键问题：

- 什么才算真正完成？
- 什么是伪完成？
- 哪些证据足够硬？
- 哪些风险必须拦截？
- 下一轮为什么应该转向？
- 什么时候应该停止？

当这些问题无法被稳定回答时，早期偏差会被后续轮次继承和加工，最终包装成一个更完整、更自洽的"完成故事"。这跟技术债的累积逻辑类似：每一轮新增的代码都在让系统看起来更完整，但底层风险从未被真正处理。

</p>

这里不是说退款安全"不能测试"。恰恰相反：越重要的部分，越应该尽量变成测试、审计记录、模拟失败、人工接管演练或其他证据。

有些判断没法变成一个分数，但可以变成一种结构：

- **优先级顺序**：真实退款路径优先于精美页面。
- **阻断条件**：越权退款、重复退款、审计缺失不能放行。
- **证据要求**：权限、资格、支付失败、客服接管都必须留下可追溯材料。
- **残余风险**：罕见支付边界可以留下，但必须看得见、被命名、有人接管。

所谓"难以 benchmark 化，但可被结构化"，意思不是放弃证明，而是承认有些证明无法压缩为一个分数，只能变为一组约束后续行动与最终评估的判断。

## 3. 同一个退款任务，如果交给 Loopora

把这个退款流程拿到工程评审会上，没人会在乎第一轮交付长得像不像正式产品。评审真正关心的是：权限路径是否可靠，资格判断有没有接入真实业务规则，支付失败后能不能追溯到记录，越权退款的风险拦住了没有，以及那些还没证明的缺口是否已经被安排进下一轮工作。

Loopora 没有复制任何工程团队的组织架构，它学的是组织架构底下那层更硬的管理约束：交付不能只靠"我自己说做完了"，风险必须经过门禁，证据必须能追溯，没证明的工作不能被包装成已完成。

Human-shaped Loop 把这套约束变成 Agent 可以持续跑下去的任务结构，让人的判断在执行过程中一直生效，而不是只在开头多叮嘱几句。

把同样的退款任务交给 Loopora，运行模式会有明显变化。

  <img src="./assets/diagrams/refund-evidence-loop.zh.svg" alt="退款任务在 Loopora 中被证据缺口牵引到下一轮" width="1000" />
</p>

**循环开始前——把需要反复纠偏的判断整理成结构规则。**

用户无需手写大量配置。Loopora 会将这些判断整理为可运行结构：

- 页面提交不是完成。
- 权限、资格、支付失败、审计和客服接管必须有证据。
- 越权退款、重复退款、审计缺失必须拦截。
- 罕见支付服务商边界可以作为残余风险，但必须看得见、被命名、有人接管。

**Agent 完成第一轮执行后，不能只汇报"我做完了"。**

假设本轮仍然交回了页面、表单、模拟资格判断和主路径测试。在普通流程中，Agent 很可能总结说：

> 页面和退款申请流程已完成，测试通过。

但在 Loopora 中，这样回答是不够的。Agent 必须按照已明确的判断标准逐项说明证据，不能临时换成一套更容易满足的问题，而要回答：

- 本轮到底证明了什么？
- 授权管理员路径有没有证据？
- 退款资格边界是真业务路径，还是模拟规则？
- 支付失败后是否有记录、账务状态和客服接管？
- 审计材料能不能让客服、财务或合规在事后还原？

于是本轮结果可能被整理如下：

| 评估面 | 本轮实际情况 |
| --- | --- |
| 已证明 | 页面可提交，主路径测试通过 |
| 弱证据 | 退款资格仍主要来自模拟规则 |
| 未证明 | 授权管理员路径、支付失败处理、审计链路 |
| 阻断风险 | 如果越权退款路径没有证明安全，就不能收尾 |

**若证据不足：下一轮将被拉回缺口。**

此时下一轮不会自由发挥，也不会让 Agent 继续打磨 UI 或补充更多提示文案。下一轮将被拉回更关键的问题上：

- 先补权限证明。
- 先补退款资格边界。
- 先补支付失败路径。
- 先补审计记录和人工接管。

Loopora 让 Agent 的自我审查不再是随意的。它不问"还能做什么让结果更完整"，而是问"我是否已按照判断标准证明了应证明的内容；若没有，下一轮必须补哪里"。

**收尾阶段不追求零风险，但结论必须诚实、可裁决。**

好的结果不一定是"所有风险都已消失"。它应清楚区分：

- **已证明**：哪些路径、边界和失败处理已经有证据撑腰。
- **未覆盖**：哪些场景还没测到。
- **阻断**：哪些问题一旦存在就不能收尾。
- **残余风险**：哪些尾巴可以带着走，但必须看得见、有人管、有后续处理方案。

Loopora 的特色在于，它并未将 Agent 包装成孤立的"天才"，而是让 Agent 的运行方式更接近工程团队。交付不能只靠自述，风险必须设卡，证据必须可追溯，未证明的工作不能被包装为完成。

自动产物仍可能出错。Loopora 要改变的是错误进入系统后的走向：它应当更早暴露，更难被粉饰过的总结掩盖；若必须带着风险前进，风险也应被命名、被看见、被评估、有人接管。

## 4. "编译"之后，判断以什么形式存在？

Loopora 常用"编译"这个词，但它不能只是个好听的比喻。

编译不是把人话翻译成更长的 prompt，而是把判断嵌入若干运行维度，让它在任务过程中持续生效。

在工程现场，这种转换每天都在发生。评审会上有人说"别只做页面，先证明权限和审计"，这句话不会只留在会议纪要里。它会变成设计约束、测试计划、上线门禁、审计检查项、回滚条件和客服接管说明。判断不是靠大家"记住"，而是被放到后续工作绕不开的地方。

这跟一次需求评审后的任务拆解很像：评审会上的一句话，最终会变成 JIRA ticket、测试计划、CI 门禁规则、上线 checklist 和 on-call 交接文档。Loopora 所说的编译也是这个意思：将一次对话中的判断，整理为后续每一轮都会遇到的运行结构。

  <img src="./assets/diagrams/judgment-surfaces.zh.svg" alt="人的判断被整理成任务契约、执行策略、证据路径和裁决规则" width="1000" />
</p>

| 人类原话 | 落入的运行维度 | 执行中如何生效 |
| --- | --- | --- |
| "别只给我一个能点的页面" | 任务契约 | 页面能点不等于做完，真实退款业务路径必须跑通 |
| "先证明权限和资格" | 证据路径 | 每轮必须说明权限、资格和边界是否有证据 |
| "越权退款绝对不行" | 裁决规则 | 一旦发现越权、重复退款或审计缺失，就不能结束 |
| "先别美化 UI" | 执行策略 | 下一轮优先补证据、修根因，而不是继续打磨界面 |
| "有些尾巴可以留" | 残余风险 | 缺口可以带着走，但必须显式声明、看得见、有人接手 |

具体说，判断进入四个用户可理解的运行维度：

- **任务契约**：什么算完成，什么是伪完成，什么必须拦截。
- **执行策略**：下一轮该优先构建、取证、修复还是收窄范围。
- **证据路径**：每轮必须留下哪些材料，证明了什么、还没证明什么。
- **裁决规则**：什么时候可以收尾，什么时候必须停下来，或者带着残余风险继续。

这几个维度并非为了显得复杂。它们只是将工程中原有的控制点压缩到 Loop 中：需求边界防止目标缩水，执行策略防止只做容易的部分，证据路径防止口头自信替代证明，裁决规则防止"看起来已完成"被包装为可交付。

Prompt 只能提醒模型。Loopora 则要将提醒转化为执行过程中可追问、可记录、可裁决的东西。

## 5. 什么时候适合 Loopora？

Loopora 不适合所有复杂任务。任务复杂与否不是决定是否使用的理由，真正决定因素在于这套判断是否需要反复执行。

工程流程有成本，不会平均分摊到每件事上。修改按钮文案不需要设计评审、灰度门禁和复盘；修复一个堆栈清楚的小 bug，通常也不需要。将重流程套在轻任务上，只会拖慢工作。

但退款、账单权限、支付回调、数据迁移这类任务不一样。设计要讲清风险，测试要证明关键边界，上线前要有人裁决，失败路径要可追踪。这些任务的风险会跨轮次积累，证据需要保留，完成不能只靠实现者自己说。

Loopora 也遵循同样的取舍。它不是复杂任务专用工具，而是当任务需要反复判断证据、风险、方向和收尾时，才值得引入的一层结构。

可按以下顺序判断：

| 判断门槛 | 如果倾向"是" | 如果倾向"否" |
| --- | --- | --- |
| 一次 Agent 执行 + 一次人工 review 够不够？ | 无需 Loopora，直接做更划算 | 继续判断 |
| 后续轮次会不会产生新证据？ | 继续判断 | 别开 Loop，只会拉长叙事 |
| 判断能不能稳定 benchmark 化？ | 优先上 benchmark 或 proof harness | 继续判断 |
| 有没有伪完成风险？ | Loopora 更值得 | 简单循环或直接 Agent 也许就够了 |
| 这套判断是否需要超出单次对话？ | 值得编译成 Loop | 直接聊就行 |

几个更具体的例子：

- **通常不需要**：生成 30 个活动主题、修一个堆栈清楚的按钮报错、拆一个边界明确的小工具函数。
- **更适合**：退款自助流程、账单权限重构、跨服务偶发支付回调丢失、需要多轮探索但又要避免套路化的品牌活动。
- **关键差别**：不是任务听起来复杂，而是人类是否会在关键轮次后反复回来判断证据、风险、方向和收尾。

## 6. 更强的模型可以解决判断力问题吗？

模型应当掌握通用能力：语言、代码、规划、工具使用、推理和审美。这些能力应当跨用户、跨任务复用。

模型越强，许多事情自然会变得更简单。它如同更资深的工程师，能提前发现更多风险，写出更好的第一版方案，减少低级错误。

但没人会因为工程师资深就取消代码评审、测试、上线门禁、审计记录和事故复盘。这和对个人能力是否信任无关，交付判断本来就不只存在于个人能力之中。哪些风险可接受、哪些证据算充分、何时能带着残余风险上线，取决于具体任务、团队承诺和业务环境，必须亮出来供大家讨论和修改。

所以，一次任务中的判断通常只能是局部的、临时的、可争议的：

- 这次退款流程必须保守，不代表所有产品任务都得保守。
- 这个原型视觉粗糙点可以接受，不代表所有原型都能这样。
- 这个 benchmark 可信，不代表所有 benchmark 都可信。
- 这次可以接受某个残余风险，不代表以后都得接受。

这些判断需要可见、可预览、可修改、可导出、可废弃。把它们放在 Agent 外部的 Loop 层，比悄悄写进模型权重或长期记忆更合适。

> 模型学习通用能力，Loop 学习当前任务的判断方式。

## 7. 最后再说哲学：偷懒来自可信自治度

未来 AI 与人类的协作，不会只沿着"模型更聪明"这一条路径演进。

模型会越来越强，但复杂任务仍然需要人类来判断：什么值得做，什么算真实完成，证据是否可信，风险是否可接受，何时继续、停止或换方向。

这套哲学不是从 AI 理论里凭空长出来的。它来自工程团队里每天都在用的东西：评审、门禁、证据、追溯、复盘，以及对"看起来完成"的天然不信任。Loopora 没有复制组织架构，只是把这套约束压缩成 Agent 能执行的任务结构。

高阶协作不是每一步都把人拽回来，也不是假装人可以完全撒手。它是让人类判断力在更合适的时间点介入任务。

Human-in-the-loop 把人类放在执行过程里。

Human-shaped Loop 把人类判断提前塑造成执行结构。

Loopora 想提升的不是 Agent 的自由度，而是可信的自治程度。自治不等于在无约束状态下持续运行，而是在人类判断的结构中持续运行。

判断、证据、转向和收尾都被外化之后，人类才能真正减少介入次数。Loopora 意义上的偷懒，不是靠牺牲质量来省事，而是通过提升可信的自治程度，让同一类判断不必被人反复执行。

这就是 human-shaped Loop。
