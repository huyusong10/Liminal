# Human-Shaped Loop：为什么长期 Agent 任务需要人的判断结构

**简体中文** | [English](./HUMAN-SHAPED-LOOP.md)

本文阐述 Loopora 背后的工程思考与协作理念。想安装、运行和使用 Loopora，请参见 [README](./README.zh-CN.md)。

Loopora 的出发点很朴素：偷懒。

更准确地说，是不想一直守在电脑前，等 Agent 跑完一轮，再指出哪里不对、催它去改。

这里想省掉的并非判断本身，而是反复做同几类判断：现在算不算完成、证据是否足够、风险能不能接受、下一轮该回到哪里、是否已经可以收尾。

每次任务的具体答案都不同，但问题类型常常重复。Loopora 想做的，是把这些会反复出现的判断提前整理成一种运行结构，让 Agent 后续每一轮都绕不开它。

## 1. 一个看起来很适合 Agent 的任务

想象一家 B2B SaaS 公司，客服团队每天处理大量退款工单。

团队决定做一个退款自助流程：客户管理员打开账单页，看到某笔订单符合退款条件，即可提交退款请求，并得到明确结果。若订单存在风险，流程就转交客服处理。

这看起来很适合交给 Coding Agent：

- 有产品界面要做。
- 有业务规则要编码。
- 有测试要补。
- 有边界情况要发现。
- 工作量足够大，一轮很可能不够。

于是用户提出需求：

> 做一个退款自助流程：客户管理员可以在账单页申请符合条件的订单退款；有风险的订单转交客服。做得安全一点，补上测试，迭代到可以作为正式功能交付为止。

第一轮结果看起来不错：页面有了，表单有了，状态提示有了，退款资格判断用了几组模拟规则，主路径测试也都通过了。

Agent 回复：

> 已经完成了！我达成了目标！

如果只是 demo，故事或许到此为止。但若作为正式产品上线，真正的问题才刚刚开始：

- 它并未证明只有授权客户管理员可以发起退款。
- 退款资格判断只是几组模拟规则，并非可靠的业务路径。
- 主路径测试通过了，但部分退款、争议订单、拒付、超出退款窗口、财务关账等情形没有被覆盖。
- 支付服务商执行失败时，系统记录、账务状态和客服接管路径并不清楚。
- 审计日志是否足以让客服、财务或合规在事后还原全过程，也没有证据。

此时，审阅者面对的不是抽象问题，而是一个具体的上线抉择：这个功能现在能否交付？

答案是还不能。

于是审阅者表示：

> 这还不能交付。先证明权限、退款资格、支付失败和审计链路，再考虑继续打磨 UI。

第二轮结果也可能看起来更像正式产品。它补了更多页面状态、更多确认文案、更多模拟数据，也在总结里提到"权限""资格""审计"这些词。

问题恰恰出在这里：人类的提醒没错，Agent 也并非完全没有采纳。但在普通对话中，这些判断仍然只是一段静态文本。它没有变成后续轮次必须引用的证据要求、不可放行的阻断条件、下一轮优先级，以及阻止收尾的终止条件。

因此 Agent 可能局部回应了提醒，却很快退回更容易完成的工作：把权限写进文案，把资格判断继续留在模拟规则里，为主路径补更多测试，然后用更自信的语气称"安全性已加强"。它做了不少事，核心风险却几乎没有减少。危险并未消失，只是被更像产品的界面和更圆满的叙事包装掩盖了。

如果继续沿这个方向多跑几轮，偏差未必表现为明显离题。第二轮把权限提示写进页面，但真实授权路径仍未证明；第三轮围绕模拟资格规则补主路径测试，却没有接入真实业务边界；第四轮整理最终说明和验收摘要，却仍说不清支付失败、账务记录和客服接管。每一轮都像在前进，但前进的仅仅是可展示的部分，不是上线门禁真正关心的退款安全。

Agent 并非不够勤奋，轮次也不算少，人类的提醒本身也没错。真正的问题在于：正确的判断没有被固化成一种能在后续轮次中持续生效的结构。

## 2. 人类介入，其实是在给循环塑形

长期 Agent 任务天然就是一个循环：执行、汇报、判断、转向，再执行。

在普通对话里，这个循环的形状主要来自 Agent 当下的理解、聊天记忆和它刚刚做过的事。人每次介入，其实是在临时改变这个循环的形状：拒绝什么、信任什么、拦截什么、下一步改什么、何时收尾。

这些介入在工程团队里通常不会只停留成一句提醒。评审会上有人说"别只做页面，先证明权限和审计"，这句话最后会落到设计约束、测试计划、上线门禁、审计检查项、回滚条件和客服接管说明里。判断不是靠大家"记住"，而是被放到后续工作绕不开的地方。

如果把人类反复做的事抽象出来，常见控制信号并不多：

| 人类反复做的事 | 稳定含义 | 工程里通常会变成 |
| --- | --- | --- |
| 说明"不算完成" | 这看起来完成，但不满足交付标准。 | 完成标准、反例、验收说明 |
| 判断"证据够不够" | 这类材料可信，那类只是自述。 | 测试结果、日志、审计记录、可追溯产物 |
| 要求"先补这里" | 下一轮不要继续铺开，先补关键缺口。 | 优先级、下一步计划、交接清单 |
| 设下"不能放行" | 某个风险存在时不能包装成完成。 | 上线门禁、回滚条件、阻断问题 |
| 决定"能否收尾" | 哪些已证明，哪些是显式残余风险。 | 验收记录、残余风险清单、后续负责人 |

这正是 Human-shaped Loop 的核心理念。

它不是把人类放回每一步执行里，也不是让人类在开头写更多话，而是把人的判断提前塑造成后续循环的形状。这个形状会决定：什么结果会被接受，什么证据足够硬，什么风险必须拦截，下一轮为什么要转向，以及任务如何诚实收尾。

所以，Human-shaped Loop 不是一个更长的提示词，也不是"多让模型反省几轮"。它关注的是：人的判断能否被预览、执行、取证、追溯和裁决。

## 3. 为什么 PRD、测试和普通循环还不够？

任务开始前问得更充分、拆得更清楚、写一份好 PRD，再让 Agent 去规划、执行、自我检查，结果通常会好很多。

这些方法都值得采用。前期澄清能降低第一轮跑偏概率；PRD 可以阐明目标、约束和范围；清单可提醒 Agent 不要遗漏常见风险；测试、类型检查、静态检查、证明脚本和基准评测能将一部分判断转化为硬性反馈；多 Agent 审查也能增加怀疑视角。

但它们主要改善的是开场质量，或者某类已经能被工具表达清楚的边界。它们无法替代执行过程中的判断回调。

真实工程团队也不会因为设计文档写得很细，就取消代码评审、测试、上线门禁、监控和复盘。文档阐明意图，执行产生新事实。退款任务进入多轮执行后，每一轮都会暴露新的问题：

- 它实际改了哪些代码和流程？
- 它绕过了哪些难点？
- 它新增的测试是在证明核心风险，还是只证明更容易通过的路径？
- 它的总结是否将"尚未证明"写成了"已经完成"？

这些事实只有在执行中才会出现，因此每一轮都必须回答：这一轮证明了什么，哪些缺口会阻断收尾，哪些风险可以被明确保留，下一轮应该扩展范围、补充证据、收窄范围、修复根因，还是停止。

**PRD / prompt 与 Human-shaped Loop 的差别在这里：**

| PRD / prompt | Human-shaped Loop |
| --- | --- |
| 描述任务开始前已知的目标与约束 | 将判断转化为执行中持续生效的控制结构 |
| 提醒 Agent 应注意哪些事项 | 要求每轮以证据回应这些事项 |
| 改善第一轮质量 | 控制多轮之间的误差传播 |
| 可能被选择性引用或局部满足 | 记录缺口、阻断问题与残余风险 |
| 主要回答"要做什么" | 持续回答"是否已证明、是否该转向、能否收尾" |

具体用例和自动检查同样重要。退款任务里，只有授权管理员能退款、超出退款窗口不能退款、重复退款必须被拒绝、支付失败必须留下记录并进入接管路径，这些都应该尽量写成稳定检查。

原则很简单：能稳定写成测试、schema、lint、类型检查、基准评测或证明脚本的判断，应优先写成这些硬证据。

但证据也有强弱。模型的自然语言总结可以帮助理解，但不能作为证据链的最终依据。

| 证据来源 | 应如何看待 |
| --- | --- |
| 测试、CI、基准评测、证明脚本、真实外部探针 | 最强，适合证明稳定契约和可机器裁决的边界。 |
| 可追溯产物、日志、截图、结构化检查结果 | 有价值，但必须说明它具体证明了什么。 |
| 独立检查或人工审阅判断 | 能暴露风险和缺口，但通常还要继续落到更具体证据。 |
| Agent 的自然语言总结 | 只能帮助阅读，不能单独作为通过依据。 |

测试只能说明某组检查通过或失败；更上层的判断还要追问：这些检查是否覆盖了本任务的完成标准，缺失部分是否阻断收尾，下一轮是否应该被拉回某个证据缺口。

没有这种门禁的循环，跑再多轮也可能放大早期偏差。Agent 会继续行动、继续总结、继续让结果看起来更完整，但底层完成定义已经偏了。

这里不是说退款安全"不能测试"。恰恰相反：越重要的部分，越应该尽量变成测试、审计记录、模拟失败、人工接管演练或其他证据。

只是有些判断无法压缩成一个分数，却可以变成结构：

- **优先级顺序**：真实退款路径优先于精美页面。
- **阻断条件**：越权退款、重复退款、审计缺失不能放行。
- **证据要求**：权限、资格、支付失败、客服接管都必须留下可追溯材料。
- **残余风险**：罕见支付边界可以留下，但必须看得见、被命名、有人接管。

这不是放弃证明，而是承认一部分证明不是单一分数，而是一组约束后续行动与最终评估的判断。

## 4. 为什么不是固定团队模板？

还有一种常见思路，是把人类团队的分工固定成一套 Agent 工作流：产品经理先分析需求，架构师做设计，工程师实现，测试或 QA 审查，最后由某个审核者签字。

这种方式有价值。真实团队确实依靠角色分工、阶段交接和标准作业流程降低混乱；当任务类型稳定、失败模式稳定、交付物也稳定时，一套固定流程可以减少无效对话，让 Agent 更有纪律。

但它解决的是另一层问题。固定团队模板主要回答：

> 谁先做，谁后做，谁审查谁？

Human-shaped Loop 要回答的是：

> 这一次任务里，什么不算完成？什么证据可信？什么风险必须阻断？下一轮应该被哪个缺口牵引？什么时候可以诚实收尾？

这两个问题不能互相替代。退款任务里，"QA" 如果只是一个通用角色名，它可能检查页面是否可用、测试是否通过、文案是否完整，却仍然没有证明授权路径、退款资格、支付失败和审计链路。数据迁移任务里的 QA 又会完全不同：它可能更应该卡幂等性、回滚、数据对账和灰度边界。一次重构任务里的审查者，则可能要优先判断行为兼容、复杂度是否真的下降、旧测试是否仍然保护公开契约。

同样叫审核者、测试者或产品经理，真正有价值的判断会随任务改变。固定模板只能提供通用分工，不能自动知道这次任务最怕哪一种伪完成。

因此，Loopora 需要动态生成 Loop 结构。这里的"动态"不是运行时随意改规则，而是在运行前根据当前任务的目标、假完成模式、证据偏好、阻断风险、执行优先级和残余风险策略，生成一份可审查的 Loop。人确认后，这份 Loop 反而应当在执行中保持稳定：后续轮次不能偷偷降低完成标准，也不能因为某个通用角色说"看起来不错"就放过必需证据。

这也是为什么 Loopora 不应退化成角色堆叠。多几个角色并不会天然更稳；只有当新增角色承担了新的证据责任、交接边界或裁决输入时，它才有意义。否则，"产品经理 + 架构师 + 工程师 + QA" 只是更像团队的叙事，不是更可信的任务判断。

可以把差异压缩成一句话：

> 固定团队模板模仿的是团队分工；Loopora 动态生成的是任务判断。前者回答"谁来做"，后者回答"这一轮凭什么算做对"。

所以，好的 Loop 不一定更复杂。它应该是能覆盖本次任务关键判断的最小结构：能用自动测试证明的地方，交给测试；需要独立取证的地方，给出证据责任；需要阻断的风险，进入裁决规则；判断不足时，先追问、审查或拒绝运行，而不是套一个万能工作流。

## 5. 在 Loopora 中落地，会发生什么？

把 Human-shaped Loop 落到 Loopora，它首先表现为三个读者可以直接感知的维度：

- **运行前可审查**：用户能看懂这套 Loop 会拒绝什么、信任什么、优先补什么、何时阻断、怎样收尾。
- **执行中可继承**：后续每一轮都拿到同一套判断、行动边界、证据缺口和输出要求，而不是只沿着聊天记忆继续。
- **收尾时可复盘**：结果能回答哪些已经证明，哪些只是弱证据，哪些仍未证明，哪些问题阻断收尾，哪些风险可以明确保留。

读者不需要先理解 Loopora 的内部名词。重要的是：判断不能只停在一段提示词或总结里，它必须变成后续工作会反复遇到的材料。

回到退款任务，运行开始前，用户无需手写大量配置。Loopora 要做的是把这些判断整理成可运行结构：

- 页面提交不是完成。
- 权限、资格、支付失败、审计和客服接管必须有证据。
- 越权退款、重复退款、审计缺失必须拦截。
- 罕见支付服务商边界可以作为残余风险，但必须看得见、被命名、有人接管。

Agent 完成第一轮后，不能只汇报一句"我做完了"。如果本轮仍然交回页面、表单、模拟资格判断和主路径测试，它必须回到已明确的判断标准：

- 本轮到底证明了什么？
- 授权管理员路径有没有证据？
- 退款资格边界是真业务路径，还是模拟规则？
- 支付失败后是否有记录、账务状态和客服接管？
- 审计材料能不能让客服、财务或合规在事后还原？

于是本轮结果可能被整理成：

| 评估面 | 本轮实际情况 |
| --- | --- |
| 已证明 | 页面可提交，主路径测试通过 |
| 弱证据 | 退款资格仍主要来自模拟规则 |
| 未证明 | 授权管理员路径、支付失败处理、审计链路 |
| 阻断风险 | 越权退款路径没有证明安全时，不能收尾 |

  <img src="./assets/diagrams/refund-evidence-loop.zh.svg" alt="退款任务在 Loopora 中被证据缺口牵引到下一轮" width="1000" />
</p>

证据不足时，下一轮不会自由发挥，也不会继续打磨 UI 或补充更多提示文案。它会被拉回更关键的问题：先补权限证明，先补退款资格边界，先补支付失败路径，先补审计记录和人工接管。

收尾阶段也不追求零风险。好的结论应该清楚区分：哪些已经证明，哪些场景还没覆盖，哪些问题会阻断收尾，哪些风险可以随行，但必须可见、有人负责、有后续处理方案。

## 6. 这给系统提出了什么要求？

Human-shaped Loop 不只是这篇文档的名字，而是一组系统必须满足的最低条件。

- 候选 Loop 不能只是任务摘要，必须携带本任务的判断边界。
- Loop 结构不能只是固定团队模板，角色和流程必须承载本任务的证据责任、交接边界和裁决输入。
- 判断不足时，系统应该追问、审查或拒绝进入可运行状态，而不是替用户编造。
- 运行开始后，每一步都应继承这些判断、行动边界和证据缺口。
- 每轮结果必须回到证据、覆盖情况、交接材料和缺口，而不是只保留一段漂亮总结。
- 最终通过必须依赖支持证据；必需证据缺失时，任务不能被包装成通过。
- 系统生命周期与任务语义必须分开：系统可以跑完，任务仍可能未被证明。

如果这些条件在产品体验里不可见，Human-shaped Loop 就只剩下漂亮说法，而不是可信的运行边界。

这也是 Loopora 与普通提示词的分界：提示词可以提醒模型，Loopora 要把提醒变成执行过程中可追问、可记录、可裁决的东西。

## 7. 什么任务适合 Loopora？

Loopora 不适合所有复杂任务。任务复杂与否不是决定因素，真正决定因素是这套判断是否需要反复执行。

工程流程有成本，不会平均分摊到每件事上。修改按钮文案不需要设计评审、灰度门禁和复盘；修复一个堆栈清晰的小 bug，通常也不需要。将重流程套在轻任务上，只会拖慢工作。

但退款、账单权限、支付回调、数据迁移这类任务不一样。设计要讲清风险，测试要证明关键边界，上线前要有人裁决，失败路径要可追踪。这些任务的风险会跨轮次积累，证据需要保留，完成不能只靠实现者自己说。

可按以下顺序判断：

| 判断门槛 | 如果倾向"是" | 如果倾向"否" |
| --- | --- | --- |
| 一次 Agent 执行加一次人工审阅够不够？ | 无需 Loopora，直接做更划算 | 继续判断 |
| 后续轮次会不会产生新证据？ | 继续判断 | 别开 Loop，只会拉长叙事 |
| 判断能不能稳定变成自动检查？ | 优先上测试、基准评测或证明脚本 | 继续判断 |
| 有没有伪完成风险？ | Loopora 更值得 | 简单循环或直接 Agent 也许就够了 |
| 这套判断是否需要超出单次对话？ | 值得编译成 Loop | 直接聊就行 |

几个更具体的例子：

- **通常不需要**：生成 30 个活动主题、修一个堆栈清晰的按钮报错、拆一个边界明确的小工具函数。
- **更适合**：退款自助流程、账单权限重构、跨服务偶发支付回调丢失、需要多轮探索但又要避免套路化的品牌活动。
- **关键差别**：不是任务听起来复杂，而是人类是否会在关键轮次后反复回来判断证据、风险、方向和收尾。

## 8. 更强的模型与可信自治

模型应当掌握通用能力：语言、代码、规划、工具使用、推理和审美。这些能力应当跨用户、跨任务复用。

模型越强，许多事情自然会变得更简单。它像更资深的工程师，能提前发现更多风险，写出更好的第一版方案，减少低级错误。

但没人会因为工程师资深就取消代码评审、测试、上线门禁、审计记录和事故复盘。这与是否信任个人能力无关。交付判断本来就不只存在于个人能力之中。哪些风险可接受、哪些证据算充分、何时能带着残余风险上线，取决于具体任务、团队承诺和业务环境，必须亮出来供大家讨论和修改。

这说明，判断力并不是一个可以一次性写进模型或全局记忆里的东西。它总是绑定在这次任务、这组风险、这个团队承诺和这批证据上。

一次任务中的判断通常只能是局部的、临时的、可争议的：

- 这次退款流程必须保守，不代表所有产品任务都得保守。
- 这个原型视觉粗糙点可以接受，不代表所有原型都能这样。
- 这次基准评测可信，不代表所有基准评测都可信。
- 这次可以接受某个残余风险，不代表以后都得接受。

这些判断需要可见、可预览、可修改、可导出、可废弃。把它们放在 Agent 外部的 Loop 层，比悄悄写进模型权重或长期记忆更合适。

> 模型学习通用能力，Loop 学习当前任务的判断方式。

Loopora 想提升的不是 Agent 的自由度，而是可信的自治程度。自治不等于在无约束状态下持续运行，而是在人类判断的结构中持续运行。

这也意味着，自治必须有边界。Loopora 不会替 Agent 获取新权限，不会绕过宿主工具的安全模型，也不把"任务通过"包装成人类最终批准。它只是把行动权限、证据缺口、阻断项和残余风险亮出来，让人少盯日常循环，但更清楚地介入关键判断。

判断、证据、转向和收尾都被外化之后，人类才能真正减少介入次数。Loopora 意义上的偷懒，不是靠牺牲质量来省事，而是通过提升可信自治度，让同一类判断不必被人反复执行。

Human-in-the-loop 把人嵌入执行过程。

Human-shaped Loop 把人的判断提前塑造成执行结构。

这就是 Human-shaped Loop。
