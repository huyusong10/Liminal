# Human-Shaped Loop：为什么长期 Agent 任务需要人的判断形状

**简体中文** | [English](./HUMAN-SHAPED-LOOP.md)

本文是一篇工程博客，解释 Loopora 背后的工程思考和协作哲学。想安装、运行和使用 Loopora，请看 [README](./README.zh-CN.md)。

Loopora 的出发点很直接：偷懒。

更准确地说，是不想一直守在电脑前面，等 Agent 做完一轮，指出哪里不对，再催它赶紧改。

真正想省掉的，不是判断本身，而是把同一类判断重复做十几遍。

先给一个直观版本：Loopora 不是帮你写一份更长的 PRD，也不是让 Agent 随便自我反省几轮。它是在 Agent 外面放一层长期任务运行结构：先把这次任务应该如何判断说清楚，再让后续每一轮都带着证据回到这些判断上。

## 1. 一个看起来很适合 Agent 的任务

想象一家 B2B SaaS 公司，客服每天都要处理大量退款工单。

团队决定做一个退款自助流程：客户管理员打开账单页，看到某笔订单符合退款条件，即可提交退款请求，并得到清楚的结果。如果某个订单有风险，流程就转交给客服处理。

这看起来很适合交给 Coding Agent：

- 有产品界面要做。
- 有业务规则要编码。
- 有测试要补。
- 有边界情况要发现。
- 工作量足够大，一轮很可能不够。

于是用户说：

> 做一个退款自助流程：客户管理员可以在账单页申请符合条件的订单退款；有风险的订单转交客服。做得安全一点，补上测试，迭代到可以作为正式功能交付为止。

第一轮结果看起来不错：页面有了，表单有了，状态提示有了，退款资格判断用了几组模拟规则，主路径测试也都通过了。

Agent 说：

> 已经完成了！我达成了目标！

这个结果第一眼给人的感觉是完整。页面在，按钮在，流程也能走通。如果这只是一个 demo，故事可能到这里就结束了。

但如果要作为正式产品交付，真正的问题现在才开始：

- 它没有证明只有授权客户管理员可以发起退款。
- 退款资格判断只是几组模拟规则，不是可靠的业务路径。
- 主路径测试通过了，但没有覆盖部分退款、争议订单、拒付、超出退款窗口、财务关账这些情况。
- 它没有说明支付服务商执行失败时，系统记录、账务状态和客服接管该怎么处理。
- 它也没有证明审计日志足以让客服、财务或合规事后还原发生了什么。

这时人类 reviewer 面对的不是一个抽象问题，而是一个很具体的上线判断：这个东西现在能不能交付？

答案是还不能。

于是人类说：

> 这还不能交付。先证明权限、退款资格、支付失败和审计链路，再考虑继续打磨 UI。

第二轮结果更像产品了。它补了更多页面状态、更多确认文案、更多模拟数据、更多主路径测试。

麻烦就在这里：它做了很多事情，但关键风险几乎没有减少。它把一个站不住脚的完成故事补得更圆了。危险没有消失，只是藏进了更像产品的界面和更自信的总结里。

如果继续沿着这个方向多跑几轮，结果可能不是突然失控，而是每一轮都“合理地”往偏处走一点：第一轮把“页面能提交”当成完成，第二轮继续打磨页面，第三轮围绕容易通过的路径补测试，第四轮开始整理 README 和最终总结。结果越来越像正式功能，但退款安全仍然没有被证明。

这个故事暴露的问题不是 Agent 不勤奋，也不是轮次不够多。表层问题是缺少一次正确的人类判断介入；更深的问题是，这种介入在长期任务里会反复出现。

人类每次回来，不只是补充需求信息。他们在给系统施加控制信号：拒绝什么、相信什么、阻断什么、下一步改什么、什么时候收口。

## 2. 为什么 PRD、用例和普通循环还不够？

这里必须认真面对几个工程上的替代方案。

### 更好的 PRD 有用，但它是地图

如果一开始问得更充分、拆得更清楚、写一份好 PRD，再让 Agent plan / execute / self-review，结果通常会好很多。

充分澄清能让第一轮少走弯路。PRD 能把目标、约束和范围写清。checklist 能提醒 Agent 不要漏掉常见风险。测试和 benchmark 能把一部分判断变成硬反馈。多 Agent review 也能增加怀疑视角。

这些方法都值得用。但它们主要改善的是“开始之前的信息质量”或“某一类硬验证”。它们降低初始误差，却不等于拥有运行中的纠偏机制。

PRD 和 prompt 更像任务开始时的地图。地图再好，也不能保证执行者在每个岔路口都按正确标准转向；也不能保证执行者走偏之后，会自动暴露偏差、补齐证据、承认不能收口。

退款任务进入多轮执行后，每一轮都会产生新的事实：

- 它实际改了哪些代码和流程？
- 它绕过了哪些难点？
- 它新增的测试是在证明核心风险，还是只是在证明更容易通过的路径？
- 它的总结是否把“没有证明”写成了“已经完成”？

这些事实只有在运行中才出现，所以每一轮都要回答运行控制问题：

- 这一轮证明的是退款业务安全，还是只证明页面能提交？
- 支付失败、重复退款、越权退款这些问题是否必须阻断？
- 哪些缺口可以作为显式残余风险，哪些不能带着走？
- 下一轮应该扩展、补证据、收窄范围、修根因，还是停止？

**静态输入和运行控制的差别在这里：**

| PRD / prompt | Human-shaped Loop |
| --- | --- |
| 描述开始前已知的目标和约束 | 把约束变成运行中持续起作用的控制结构 |
| 提醒 Agent 应该注意什么 | 要求每轮拿证据回应这些注意事项 |
| 改善第一轮质量 | 控制多轮误差传播 |
| 可以被选择性引用或局部满足 | 记录缺口、阻断问题和残余风险 |
| 主要回答“要做什么” | 持续回答“是否证明了、是否该转向、是否能收口” |

### 固定用例有用，但它守的是已知边界

退款任务里，有些东西当然应该变成固定用例：

- 只有授权管理员能退款。
- 超出退款窗口不能退款。
- 重复退款必须被拒绝。
- 支付失败必须留下记录并进入接管路径。

这些用例越硬越好。Loopora 不替代它们。

但固定用例主要守住已知、稳定、可枚举的边界。长期任务里还有更上层的问题：

- 用例失败了，下一轮应该先修哪里？
- 用例没覆盖，但证据明显不足，能不能收口？
- Agent 新增的测试是在证明核心风险，还是绕开难点？
- 新出现的缺口是阻断风险，还是可见残余风险？
- 当固定用例不完整时，系统如何要求 Agent 暴露“未证明”？

> 固定用例守住已知边界；Human-shaped Loop 管住长期任务里不断出现的判断、证据和转向。

### 普通自动循环会放大早期误差

现在已经有很多方式可以让 Agent 多跑几轮：重复调用同一个 Agent、让模型自己审自己、给它 checklist 再继续、跑测试再修。

这些方法在外部反馈清楚时非常有效。benchmark、contract test、schema、lint、type check 和 proof harness 都能给出确定反馈。当判断已经被这些工具表达清楚，简单循环就够了。

但退款流程更麻烦。它的关键判断很难稳定压成一个分数。测试很重要，但主路径测试通过不等于业务流程安全。漂亮 UI 也重要，但它可能掩盖一个事实：退款权限、审计、支付服务商失败和客服接管都没有被证明。

没有治理的循环像开盲盒。它让 Agent 多次行动，却没有稳定回答：

- 什么才算真正完成？
- 什么是假完成？
- 哪些证据足够硬？
- 哪些风险必须阻断？
- 下一轮为什么应该转向？
- 什么时候应该停下？

当这些问题没有被稳定回答时，早期误差就会被后续轮次继承、加工，最后变成一个更完整、更自洽的完成故事。

<p align="center">
  <img src="./assets/diagrams/error-propagation.zh.svg" alt="普通自动循环如何把早期误差加工得更像完成" width="1000" />
</p>

这里不是说退款安全“不能测试”。恰恰相反，越重要的部分越应该尽量变成测试、审计记录、模拟失败、人工接管演练或其他证据。

关键在于：还有一部分判断不是单一分数，而是一组结构。

它可以被拆开：

- 偏序：真实退款路径优先于漂亮页面。
- 阻断条件：越权退款、重复退款、审计缺失不能放行。
- 证据要求：权限、资格、支付失败、客服接管都必须留下可追溯材料。
- 残余风险：罕见支付边界可以留下，但必须可见、被命名、有人接管。

这就是“难以 benchmark 化，但可以结构化”的意思。不是放弃证明，而是承认有些证明不是一个分数，而是一组会约束后续行动和裁决的判断。

## 3. 同一个退款任务，如果交给 Loopora

Loopora 的核心动作，是把未来反复发生的人类纠偏提前到运行之前，再让它变成可运行的控制结构。

Human-shaped Loop 不是“人提前说更多话”。更准确地说，它让人的判断在运行中继续起作用。

回到退款任务，如果交给 Loopora，运行形状会明显不同。

<p align="center">
  <img src="./assets/diagrams/refund-evidence-loop.zh.svg" alt="退款任务在 Loopora 中被证据缺口牵引到下一轮" width="1000" />
</p>

**运行前：把未来会反复纠偏的判断说清楚。**

用户不需要手写一大堆配置，但系统应该帮助用户把这些判断显影：

- 页面提交不是完成。
- 权限、资格、支付失败、审计和客服接管必须有证据。
- 越权退款、重复退款、审计缺失必须阻断。
- 罕见支付服务商边界可以作为残余风险，但必须可见、被命名、有人接管。

**第一轮之后：Agent 不能只说“我做完了”。**

假设第一轮仍然交回了页面、表单、模拟资格判断和主路径测试。普通流程里，Agent 很可能总结：

> 页面和退款申请流程已完成，测试通过。

但在 Loopora 里，这个回答还不够。Agent 需要沿着已经显影的人类判断标准交代证据。它不能临时换一套更容易满足的问题，而要回答：

- 这轮到底证明了什么？
- 授权管理员路径有没有证据？
- 退款资格边界是真业务路径，还是模拟规则？
- 支付失败后是否有记录、账务状态和客服接管？
- 审计材料能不能让客服、财务或合规事后还原？

于是第一轮结果可能会被整理成这样：

| 裁决面 | 第一轮实际情况 |
| --- | --- |
| 已证明 | 页面可提交，主路径测试通过 |
| 弱证据 | 退款资格仍主要来自模拟规则 |
| 未证明 | 授权管理员路径、支付失败处理、审计链路 |
| 阻断风险 | 如果越权退款路径没有证明安全，就不能收口 |

**如果证据不够：下一轮被压回证据缺口。**

这时下一轮不是自由继续，也不是让 Agent 继续美化 UI 或补更多确认文案。下一轮会被拉回更硬的问题：

- 先补权限证明。
- 先补退款资格边界。
- 先补支付失败路径。
- 先补审计记录和人工接管。

也就是说，Loopora 让 Agent 的“自我反省”不再是随意的。它不是问“我还能做点什么让结果更完整”，而是问“我有没有沿着人的判断标准证明该证明的东西；没有的话，下一轮必须补哪里”。

**最终收口：不承诺完全正确，但必须诚实可裁决。**

好的结果不一定是“所有风险都消失”。它应该清楚地区分：

- 已证明：哪些路径、边界和失败处理已经有证据支撑。
- 未证明：哪些场景还没有覆盖。
- 阻断：哪些问题一旦存在就不能收口。
- 残余风险：哪些缺口可以带着走，但必须可见、有 owner、有后续处理方式。

这就是 Loopora 的特色：它不承诺自动产物绝对正确，但它改变错误被处理的方式。错误应该更早暴露，更难被包装成完成；如果必须带着风险前进，风险也要被命名、被看见、被裁决。

## 4. “编译”之后，判断以什么形式存在？

Loopora 常用“编译”这个词，但它不能只是一个好听的隐喻。

编译不是把人话翻译成更长 prompt，而是把判断放进几个运行面里，让它们在任务过程中持续起作用。

<p align="center">
  <img src="./assets/diagrams/judgment-surfaces.zh.svg" alt="人的判断被整理成任务契约、执行姿态、证据路径和裁决规则" width="1000" />
</p>

| 人的话 | 进入的运行面 | 运行中怎么起作用 |
| --- | --- | --- |
| “别只给我一个能点的页面” | 任务契约 | 页面能点不算完成，真实退款业务路径必须跑通 |
| “先证明权限和资格” | 证据路径 | 每轮必须说明权限、资格和边界是否有证据 |
| “越权退款绝对不行” | 裁决规则 | 一旦发现越权、重复退款或审计缺失，就不能收口 |
| “先别美化 UI” | 执行姿态 | 下一轮优先补证据和修根因，而不是继续 polish |
| “有些尾巴可以留” | 残余风险 | 缺口可以带着走，但必须显式、可见、有人接管 |

用更直接的话说，判断被放进四个用户能理解的运行面：

这里的四个面是文章里的概念压缩。README 从“方案文件是否足够可审查”的角度，会把 **执行姿态** 再拆成 Agent 责任和运行流程，于是得到五个面向阅读的判断面。两种说法指向同一件事：运行期仍然落回 Loop 契约、角色、流程和证据，裁决是它们共同形成的投影，不是另一份事实源。

- **任务契约**：什么算完成，什么是假完成，什么必须阻断。
- **执行姿态**：下一轮应该优先构建、取证、修复还是收窄。
- **证据路径**：每轮必须留下哪些材料，证明了什么、没证明什么。
- **裁决规则**：什么时候可以收口，什么时候必须停止或带残余风险继续。

Prompt 可以提醒模型。Loopora 要让提醒变成运行中可以追问、记录和裁决的东西。

## 5. 什么时候适合 Loopora？

Loopora 不适合所有复杂任务。复杂不是理由，重复判断才是理由。

可以按这个顺序判断：

| 判断门 | 如果答案偏“是” | 如果答案偏“否” |
| --- | --- | --- |
| 一次 Agent 执行 + 一次人工 review 够不够？ | 跳过 Loopora，直接做更便宜 | 继续判断 |
| 后续轮次会不会产生新证据？ | 继续判断 | 不要开 Loop，只会制造更长叙事 |
| 判断能不能稳定 benchmark 化？ | 优先 benchmark 或 proof harness | 继续判断 |
| 有没有假完成风险？ | Loopora 更有价值 | 简单循环或直接 Agent 可能够用 |
| 这套判断是否需要活过一次聊天？ | 值得编译成 Loop | 直接聊天即可 |

几个更具体的例子：

- **通常不需要**：生成 30 个活动主题、修一个堆栈清楚的按钮报错、拆一个边界明确的小工具函数。
- **更适合**：退款自助流程、账单权限重构、跨服务偶发支付回调丢失、需要多轮探索但又要避免套路化的品牌活动。
- **关键差别**：不是任务听起来复杂，而是人类是否会在关键轮次后反复回来判断证据、风险、方向和收口。

## 6. 更强的模型可以解决判断力问题吗？

模型应该学习通用能力：语言、代码、规划、工具使用、推理模式和广泛审美。这些能力应该跨用户、跨任务复用。

但一次任务里的判断通常是局部的、临时的、可争议的：

- 这次退款流程必须保守，不代表每个产品任务都要保守。
- 这个原型可以接受视觉粗糙，不代表所有原型都可以。
- 这个 benchmark 可信，不代表所有 benchmark 都可信。
- 这次可以接受某个残余风险，不代表它是长期偏好。

这些判断应该显式、可预览、可修改、可导出、可废弃。它们更适合存在于 Agent 外部的 Loop 层，而不是静默写进模型权重或长期记忆。

> 模型学习通用能力，Loop 学当前任务的判断方式。

## 7. 最后再说哲学：偷懒来自可信自治度

未来 AI 与人类的协作，不只会沿着“模型更聪明”这一条线演进。

模型会继续变强，但复杂任务仍然需要人类判断：什么值得做，什么算真实完成，证据是否可信，风险是否可接受，何时继续、停止或换方向。

真正的高阶协作，不是每一步都把人拉回，也不是假装人可以完全离开，而是让人类判断力以更合适的时间形态参与任务。

Human-in-the-loop 把人类放在执行过程里。

Human-shaped Loop 把人类判断提前塑造成执行结构。

Loopora 想提高的不是 Agent 的自由度，而是可信自治度。自治不是无人约束地继续跑，而是在人的判断结构里继续跑。

当判断、证据、转向和收口都被外化，人才可以真正少回来几次。这就是 Loopora 意义上的偷懒：不是牺牲质量来省事，而是通过提高可信自治度，让同一类判断不必被人重复执行。

这就是 human-shaped Loop。
