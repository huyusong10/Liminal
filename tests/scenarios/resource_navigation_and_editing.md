# 场景：资源库、Loop 卡片与专家编辑入口

**目标**：确认 Loop、运行、资源库、角色定义和流程编排这些入口能互相衔接，同时危险操作和主进入动作保持分离。

**前置**：

- 至少存在一条有运行的 Loop、一条无运行的 Loop、一个自定义角色和一个自定义编排。
- 页面可在中文或英文任一界面语言下打开。

**步骤**：

1. 从 Loop 页打开有运行的 Loop 卡片主体，确认进入最值得继续查看的运行或 Loop 详情。
2. 返回 Loop 页，分别触发同一卡片的查看 Loop 与删除动作，确认它们不会被整卡点击吞掉。
3. 进入 Loop 详情页，确认配置、Loop 契约、运行历史和最近摘要能帮助用户先理解 Loop，再进入具体运行。
4. 进入角色定义与流程编排列表，确认自定义卡片主体可进入编辑页，删除按钮保持独立。
5. 新建或编辑编排，确认空白起点不会自动变成默认预设，步骤卡片、流程图和设置浮窗能同步表达同一份流程。
6. 在手动编排页打开 Loop 契约编辑器，确认编辑、预览、保存和关闭围绕同一路径工作。

**预期**：

- 顶层导航保持“Loop / 编排 / 资源库 / 工具 / 教程”的主心智；Loop 工作台承接活动 Loop 和已有 Loop 高频浏览，编排工作台承接对话编排、导入方案文件和手动编排，运行列表不再作为独立一级页面。
- 卡片主体、次级入口和危险操作拥有独立可达的交互目标。
- 角色、流程、Loop 契约的专家编辑仍回到同一套 Loop / 方案包生命周期，不制造孤立资产。

**为什么保留为 scenario**：

- contract checks 能证明单个路由、API 和编辑保存契约，browser journeys 能覆盖代表性点击路径，但这个场景关注多个资源入口之间的心智连续性和危险动作分离。
- review cases 能提示页面语言或布局问题，却不能替代操作者实际穿梭 Loop 卡片、详情、资源库和专家编辑器时的路线判断。
- 这条 scenario 保留为低频人工/Agent 探索入口，用于发现“自动化都绿但导航意图变得混乱”的问题。
