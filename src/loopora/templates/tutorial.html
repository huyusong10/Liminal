按正确顺序使用 Loopora：先定义角色，再编排流程，最后创建循环。 Use Loopora in the right order: define roles first, orchestrate the flow second, then create loops.

Loopora 现在把“角色是谁、怎么执行”和“流程怎么串起来”拆开了。照着下面三步走，后面的循环会清爽很多。 Loopora now separates role identity and execution from orchestration wiring. Follow the three steps below and your loops stay much easier to reason about.

先判断这次改动属于哪一层Start by choosing the right layer

如果你一时只想知道“应该回到哪个页面去改”，先看下面这一排，能更快避开职责混淆。 If all you need first is “which page should I go back to?”, start with this row so you can avoid crossing responsibilities.

改 prompt / 工具 / 默认模型 Changing prompts, tools, or default models 先回到角色定义 Go back to Role Definitions

角色是谁、默认怎么跑、权限边界在哪里，都属于这里。 Who the role is, how it runs by default, and where its permission boundary sits all belong here.

改顺序 / 开关步骤 / 收束逻辑 Changing step order, toggles, or completion 去流程编排 Go to Orchestrations

这里决定角色怎样协作，但不改角色自身默认 prompt 和执行配置。 This is where roles collaborate, but it does not change the role's own default prompt or execution profile.

换项目 / spec / 轮次策略 Changing project, spec, or run policy 去创建循环 Go to Create Loop

workdir、spec、completion mode、轮次间隔和最大轮数，都在这层决定。 Workdir, spec, completion mode, iteration interval, and max rounds are all decided at this layer.

编排页只负责选哪些角色、每轮按什么顺序跑、哪些步骤启用，以及 GateKeeper 通过后是否结束。角色 prompt 和执行工具不在这里改。 The orchestration page only chooses roles, defines iteration order, enables or disables steps, and decides whether GateKeeper finishes the run. Prompt and executor changes do not happen here.

关键原则Key principles

角色定义回答“这个角色是谁、能做什么、默认怎么跑”。流程编排回答“这些角色怎样协作”。创建循环回答“这套流程在这个项目里怎么持续运行”。 Role definitions answer “who is this role and how does it run by default”. Orchestrations answer “how do these roles collaborate”. Loop creation answers “how should this collaboration keep running in this project”.

如果你发现自己想在编排里改 prompt 或执行工具，通常说明这项改动应该回到“角色定义”。如果你只是想调整运行频率、最大轮数或 completion mode，那就是“创建循环”的职责。 If you find yourself trying to change prompts or executors inside an orchestration, that usually means the change belongs back in Role Definitions. If you only want to tune cadence, max rounds, or completion mode, that belongs in Create Loop.

上下文如何在流程里流转How context moves through a workflow

Loopora 不会把每一步都变成一段“自由发挥”的 prompt 拼接。运行开始时会先冻结契约，再在每个步骤前后写出固定结构的 context 与 handoff 文件，让下一步和下一轮都能按代码规则继续消费。 Loopora does not rely on free-form prompt drift between steps. It freezes the contract first, then writes structured context and handoff files around every step so the next step and the next round can consume them by code-defined rules.

1. 运行开始1. Run start

冻结全局契约Freeze the run contract

spec、workflow、prompt 引用和运行策略会先被收敛成稳定快照，后面的步骤都以这份契约为准。 The spec, workflow, prompt refs, and runtime policy are frozen into one stable snapshot that every later step reads from.

contract/run_contract.json
2. 步骤开始前2. Before a step

装配步骤上下文Assemble step context

当前轮次、当前角色、紧邻上一步 handoff、上一轮同 step 摘要和可用 artifacts 会被整理成一个稳定包。 The current iteration, current role, immediate previous handoff, prior same-step summary, and available artifacts are assembled into one stable packet.

iterations/iter_000/steps/00__builder/input.context.json
3. 步骤结束后3. After a step

写回输出与交接Write output and handoff

角色原始输出会先归档，再由系统代码提炼出 handoff，交给下一个步骤和下一轮继续使用。 The role's raw output is archived first, then Loopora derives a handoff that the next step and the next iteration can reuse.

iterations/iter_000/steps/00__builder/handoff.json
4. 每轮收束4. Iteration close

生成轮次总结Generate iteration summary

每轮会落一份总结，记录关键 handoff、score、stagnation 和最新索引，供下一轮直接承接。 Each round writes a summary with the key handoffs, score, stagnation state, and latest indexes so the next round can pick up cleanly.

context/latest_iteration_summary.json

下一步到底会读什么What the next step actually reads

  • 先读固定契约：`contract/run_contract.json` 提供 spec、workflow 和运行边界。 It starts with the fixed contract: `contract/run_contract.json` provides the spec, workflow, and runtime boundaries.
  • 再读当前 step 的 `input.context.json`，里面已经明确写好“现在是第几轮、上一轮发生了什么、紧邻上一步是谁”。 Then it reads the current step's `input.context.json`, which already states the iteration number, what happened last round, and who ran immediately before it.
  • 如果上一步产生了 `handoff.json`，当前步骤会优先消费它，而不是自己去猜上一位角色做了什么。 If the previous step wrote a `handoff.json`, the current step consumes that first instead of guessing what the prior role did.
  • 跨轮次时，会继续参考 `iterations/iter_000/summary.json` 和 `context/latest_state.json`，避免历史信息散落在长 prompt 里。 Across iterations, it also consults `iterations/iter_000/summary.json` and `context/latest_state.json` so history does not sprawl across long prompts.

运行中会出现的中间文件Intermediate files you will see during a run

这些文件既是白盒观察入口，也是后续步骤继续消费的结构化产物。终端里的系统动作会告诉你它们什么时候被准备、写入和汇总。 These files are both white-box observation points and structured artifacts that later steps continue to consume. The console's system actions tell you when they are prepared, written, and summarized.
