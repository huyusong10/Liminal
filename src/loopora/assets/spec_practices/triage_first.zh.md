## 场景

hybrid search 正在从帮助中心继续扩到 API 文档和内部手册。与此同时，多类问题开始一起冒出来：结果陈旧、排序漂移、筛选失效、偶发权限漏出。团队这轮不可能把所有症状都修掉。这次 loop 的任务，是先把这团噪声压缩成一个真正值得吸收实现精力的阻塞切片。

## 需求

把这组混合 rollout 症状收敛成一个站得住证据的阻塞切片，并围绕这个切片完成实现；不要假装这一轮就能把整个搜索 rollout 稳定下来。

## 适合这个流程，因为

现在最难的不是修，而是先决定这轮到底在修什么。Inspector 要先把反复出现的真实信号和 rollout 噪声分开，Guide 要把这个诊断翻译成一个高杠杆下一步，之后 Builder 才应该动手，而 GateKeeper 也才能围绕已经澄清的目标裁决。

## 为什么不是其他流程

这不是 `Inspect First`，因为现在还没有收敛到一个稳定缺陷、一条稳定证据路径。它也不是 `Build First`，因为让 Builder 先动手，只会让实现跑在问题定义前面。更不是 `Repair Loop`，因为这一轮要解决的还是“先选哪一个阻塞切片”，而不是第二轮该怎么补。

## 为什么不直接交给 AI Agent

如果没有 Loopora，AI Agent 当然可以先选一个症状开始改，但人类随后就得不断回来确认：它选的到底是不是本轮最值得修的阻塞点。这里的瓶颈不是实现能力，而是人类反复回来重定范围、重做判断。Loopora 的意义，就是先让 `Inspector` 压缩证据、让 `Guide` 选定切片，之后再让 `Builder` 和 `GateKeeper` 把精力花在这一轮真正应该被裁决的对象上。

## 示例 spec

# Task

把这组 hybrid-search rollout 的混合症状收敛成一个已验证的阻塞切片，并围绕这个切片完成实现，同时保留更大 rollout 的上下文。

# Done When

- 当前症状云已经被收敛成一个已诊断的子系统或明确决策点，并且有直接证据支撑。
- 已经围绕这个诊断切片选定并落地一条修复方向。
- 同一条被澄清的触发路径不再出现这个阻塞，或者剩余阻塞已经收敛成一个已验证的单点缺口。
- 诊断链仍然可以通过命令、trace 或 rollout 产物回溯，供后续 loop 继续接力。

# Guardrails

- 不要把这一轮扩张成整个搜索 rollout 的稳定化重写。
- 没有被证明属于本次阻塞切片的行为应保持原样。
- 从证据、修复方向到结果，整条链路都要可追溯，便于下一轮继续推进。

# Role Notes

## Inspector Notes

先把反复出现的 rollout 信号和噪声分开，再把症状云压缩成最小、最站得住的阻塞定义。

## Guide Notes

把诊断结果翻译成一个高杠杆下一步，不要展开成整份 rollout 路线图。

## Builder Notes

只有在切片已经足够窄、GateKeeper 可以围绕直接证据裁决时，才开始实现。

## GateKeeper Notes

这一轮只对被澄清后的阻塞切片裁决，不要拿整个 rollout 来要求这一轮一次性收完。
