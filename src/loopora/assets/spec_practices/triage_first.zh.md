## 场景

客服不断收到反馈，说密码重置邮件“有时候会来两封”。这些反馈并不稳定，团队也还没确认到底是 Web UI、后端 API，还是异步邮件队列在重复触发。

## 需求

先把这个模糊投诉收敛成一个已验证的修复方向，再围绕这个方向落地实现并拿到证据。

## 适合这个流程，因为

问题定义本身还不稳定，所以应该先让 Inspector 把问题收窄，再让 Guide 把诊断翻译成一个明确动作，最后才轮到 Builder 落实现、GateKeeper 做裁决。

## 为什么不是其他流程

这不是 `Inspect First`，因为现在连失败边界都还很模糊；也不是 `Build First`，因为让 Builder 先动手，只会逼着实现跑在任务定义前面。它同样不是 `Repair Loop`，因为我们现在要找的是第一轮修复方向，而不是第二轮该怎么补。

## 示例 spec

# Task

把“密码重置邮件偶尔会发两封”的投诉收敛成一个已验证的修复方向，再围绕这个方向完成实现并拿到证据。

# Done When

- 当前问题已经被收敛成一个已诊断的层级或明确决策点，并且有直接证据支撑。
- 有一条围绕该诊断方向的实现已经落地。
- 同一触发路径不再发送重复邮件，或者剩余阻塞已经收敛成一个已验证的缺口。
- 诊断过程仍然可以通过文件、命令或产物回溯。

# Guardrails

- 在问题还没澄清前，不要把范围扩散成无关清理。
- 没有被证明属于问题的邮件流程应保持原样。
- 从证据到最终修复，整条诊断链都要可回溯。

# Role Notes

## Inspector Notes

把模糊性压缩成最小、最具体的问题描述。

## Guide Notes

把诊断结果翻译成一个高杠杆的下一步，不要展开成长篇策略报告。

## Builder Notes

只有在问题描述已经足够窄、可以验证时才开始实现。

## GateKeeper Notes

要基于澄清后的问题定义裁决，而不是回到最初模糊的投诉表述。
