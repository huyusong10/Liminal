## 场景

帮助中心内容域已经进入 shadow。大多数查询看起来还行，但一小组高价值查询在新搜索里开始出现明显退化：要么缺关键结果，要么结果明显比旧搜索更旧。团队现在还不能确认，问题第一次出现在采集 freshness、索引构建、召回、重排还是权限过滤。

## 需求

先把这组退化查询的第一个失效层钉住，再围绕那个层级完成修复；不要把这一轮扩张成一次通用搜索重写。

## 适合这个流程，因为

失败面已经知道了：有一组稳定的高价值查询在退化。现在最缺的不是代码，而是可信证据，来说明缺口第一次出在哪一层。只有 Inspector 先把这个点钉住，Builder 才知道该修哪一段；GateKeeper 也才能沿着同一条证据路径判断修复有没有推进。

## 为什么不是其他流程

这不是 `Build First`，因为帮助中心路径已经存在了，现在缺的不是第一条切片，而是证据。它也不是 `Triage First`，因为当前问题已经收敛成一组查询退化，而不是一团互相无关的症状。更不是 `Repair Loop`，因为现在还在决定第一轮该修哪一层，而不是第二轮该怎么补。

## 为什么不直接交给强 Agent

如果没有 Loopora，强 Agent 很可能会立刻去改召回、重排或 freshness 逻辑，但人类随后就得不断回来确认：它到底是不是在改对的层、是不是又沿着错假设继续推进。这里一轮 patch 之后再 review 一次是不够的，因为真正关键的问题发生在更前面：第一个可信缺口到底在哪。Loopora 的作用，就是先让 `Inspector` 把证据钉住，再让 `Builder` 围绕它修，最后让 `GateKeeper` 沿同一路径裁决，从而减少人类反复回来纠偏的次数。

## 示例 spec

# Task

定位 shadow 流量里那组高价值帮助中心查询第一次退化的层级，并在不改变更大搜索契约的前提下完成修复。

# Done When

- 一组代表性的高价值查询已经能通过项目内证据稳定复现退化。
- 退化第一次出现在哪一层，已经被直接证据钉住。
- 已经围绕这个被钉住的层级完成了一次修复。
- 同一组代表性查询在修复后得到改善，或者剩余阻塞已经被收敛成一个已验证的单点缺口。
- 最终验证沿用了最初暴露退化的那条证据路径。

# Guardrails

- 在第一个失效层还没钉住前，不要把这一轮扩张成搜索栈大重写。
- 保留现有帮助中心搜索入口和对外搜索契约。
- 保留能解释退化问题的日志、trace 和产物，避免后续 loop 重新盲查。

# Role Notes

## Inspector Notes

先钉住退化第一次出现在采集 freshness、索引、召回、重排还是权限过滤。

## Builder Notes

只围绕已经站住证据的层级动手，不要一次性补多个猜测。

## GateKeeper Notes

只有当同一组代表性查询沿同一路径得到改善时，才允许放行。

## Guide Notes

如果证据还很噪，就把下一步收敛成更窄的诊断切片，而不是泛泛地“把搜索整体整理一下”。
