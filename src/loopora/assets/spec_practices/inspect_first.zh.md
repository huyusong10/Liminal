## 场景

你刚接手一条月结账单流水。财务团队发现，少数大租户的发票归档偶尔会缺文件，但总金额看起来又是对的。现在没人能说清问题到底出在 usage 汇总、发票渲染、归档打包，还是对象存储上传重试。

## 需求

先把这条账单流水里“归档为什么会不完整”查明白，再修掉真实根因；不要趁机把整条 billing close 重写一遍。

## 适合这个流程，因为

这类任务的第一关键不是“先改一版代码”，而是先让 Inspector 把失败形态和证据链钉牢。只有证据站住了，Builder 才知道该修哪一段；GateKeeper 也才能沿着同一条证据链判断这次修复到底有没有推进。

## 为什么不是其他流程

这不是 `Build First`，因为现在最缺的不是第一版实现，而是可信证据。它也不是 `Triage First`，因为问题边界已经不算模糊，真正缺的是根因落点。更不是 `Repair Loop`，因为在第一轮该修哪里都还没钉住之前，谈第二轮修复还太早。

## 示例 spec

# Task

定位月结账单流水里“部分租户发票归档不完整”的真实根因，并在不改变现有账单关闭契约的前提下完成修复。

# Done When

- 已经通过项目内直接证据复现，或明确解释，发票归档为什么会缺文件。
- 至少一个真实根因已经被验证并修复，或者剩余阻塞已经被收敛成一个已验证的单点缺口。
- 同一条月结流水现在能生成完整归档，或者只留下一个更小、已确认的残余问题。
- 最终验证沿用了最初暴露问题的那条证据路径，而不是换了一套新的解释。

# Guardrails

- 在拿到完整证据链之前，不要先做 billing close 的大范围重构。
- 保留现有账单关闭、归档命名和对外可见的交付契约。
- 保留能解释问题的日志、报告和中间产物，避免第二轮分析重新盲查。

# Role Notes

## Inspector Notes

先把哪一类租户、哪一轮 close、哪一步产物开始缺失这三件事查清，再推动代码修改。

## Builder Notes

只围绕已经站住证据的根因动手，不要顺手把 usage 汇总、归档打包和上传重试一起重写。

## GateKeeper Notes

只有当同一条月结流水的归档完整性被重新验证通过，才允许放行。

## Guide Notes

如果证据还很散，就先把下一步收敛成更窄的诊断切片，而不是泛泛地建议“把整条流水整理一下”。
