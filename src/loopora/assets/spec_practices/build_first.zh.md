## 场景

一个知识库产品正在把旧的 keyword-only 搜索升级成新的 hybrid search。上线不是一次性全切，而是先从一个明确的内容域开始：公开帮助中心。这次 loop 并不负责整个搜索升级项目，它只负责把帮助中心这一段真正跑通，好让团队可以基于真实内容决定它能不能进入 shadow，而不是继续停留在架构讨论里。

## 需求

把帮助中心内容域的第一条 hybrid-search 端到端路径跑通：采集、索引、检索、权限过滤和搜索页展示都要真正接起来；同时保留现有搜索入口，并为 shadow 留下清晰的 go / no-go 判断。

## 适合这个流程，因为

目标已经很清楚。现在最缺的不是更多诊断，而是一条 Builder 真正能跑出来的完整路径。只有这条路径存在了，Inspector 才能在真实帮助中心内容上验证，GateKeeper 才能判断它是否已经够资格进入 shadow。只有卡住时才需要 Guide 帮忙收窄下一步。

## 为什么不是其他流程

这不是 `Inspect First`，因为团队现在不是卡在一个神秘缺陷上，而是卡在第一条真实路径还没建立起来。它也不是 `Triage First`，因为这一轮范围已经很明确。更不是 `Repair Loop`，因为在基线路径还没出现之前，还谈不上第二轮修复的问题。

## 为什么不直接交给 AI Agent

AI Agent 当然可以先把第一版代码写出来。但如果没有 Loopora，人类就得在采集接上之后、索引跑起来之后、第一批查询出来之后反复回来确认同一类问题：这条路径现在到底算不算完整、结果够不够进入 shadow、接下来最该补的是哪一层。Loopora 的价值，就是把这些原本会反复占用人类注意力的确认点，交给 `Builder`、`Inspector`、`GateKeeper` 和 `Guide` 按顺序消化掉，让人类只在拿到新的证据和裁决之后再介入。

## 示例 spec

# Task

交付帮助中心内容域的第一条 hybrid-search 切片，让现有搜索入口能够通过新链路服务真实帮助中心内容，并留下一个可信的 shadow 判断。

# Done When

- 帮助中心内容可以端到端走通新的采集与索引路径。
- 现有搜索入口可以从新的帮助中心搜索链路读取结果。
- 至少一组项目内定义的代表性帮助中心查询，可以通过新链路返回可用结果。
- 权限过滤和基础结果展示在这条服务路径上保持成立。
- 当前证据已经足够让 `GateKeeper` 判断帮助中心切片能不能进入 shadow。

# Guardrails

- 这一轮只聚焦帮助中心切片，不要扩张成整个搜索升级项目。
- 复用现有搜索入口和公开搜索页，不要另起一套平行搜索界面。
- 在引入新切片期间，保留现有对外搜索契约。

# Role Notes

## Builder Notes

优先把第一条可进入 shadow 的真实路径落出来，不要把时间花在整个迁移外围的推测性清理上。

## Inspector Notes

先在真实帮助中心内容上验证采集、索引、检索、过滤和搜索页展示，再看次要打磨。

## GateKeeper Notes

只有当帮助中心切片已经足够支撑 shadow 判断时，才允许放行。

## Guide Notes

如果进展卡住，就把下一步收敛到阻止 shadow 判断成立的那一条缺口上。
