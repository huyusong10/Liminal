## 场景

平台团队已经拍板要做一个新的 workspace 导入流。管理员应该可以上传一个项目归档包，让系统把它解包到托管存储里，然后在现有项目列表里看到这个新导入的 workspace。

## 需求

把这个新的 workspace 导入路径端到端交付出来，但不要重做现有项目列表，也不要带坏现在的手动创建流程。

## 适合这个流程，因为

目标已经很清楚，先让 Builder 把第一版真实路径跑起来最有效。随后再由 Inspector 补证据、GateKeeper 判断现有入口有没有被带坏，只有推进卡住时才需要 Guide 出来指方向。

## 为什么不是其他流程

这不是 `Inspect First`，因为团队现在并不缺“问题到底在哪”的证据；也不是 `Triage First`，因为任务本身已经定义得很清楚。它同样不是 `Repair Loop`，因为我们并不是带着“一轮做完马上还要再修一轮”的预期进来的。

## 示例 spec

# Task

交付新的 workspace 导入流，让管理员可以上传项目归档包、完成导入，并在现有项目列表里看到导入后的 workspace。

# Done When

- 管理员可以从现有导入入口上传一份受支持的项目归档包。
- 归档包会被成功解包到托管存储里，并创建新的 workspace 记录。
- 导入后的 workspace 会以预期元数据出现在现有项目列表里。
- 有一条项目内检查或可复现的本地验证，证明整条导入路径在当前工作区真正可用。
- 现有手动创建 workspace 和项目列表入口继续正常工作。

# Guardrails

- 复用当前项目列表和导入入口，不要再造一条平行流程。
- 保留现有手动创建 workspace 和项目列表路由。
- 改动范围只聚焦在导入路径和导入结果的可见性。

# Role Notes

## Builder Notes

优先尽快落出真实可用的端到端导入路径，不要把整轮时间都花在诊断上。

## Inspector Notes

先核对上传、解包和导入结果可见性，再去看次要 polish。

## GateKeeper Notes

只有当导入在当前工作区真正可用，且原有项目入口没有被带坏时，才允许放行。

## Guide Notes

如果进展卡住，就把下一步收敛到 `Done When` 里最缺失的一条。
