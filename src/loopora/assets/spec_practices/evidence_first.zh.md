## 场景

一个长期任务已经有可见症状或宽泛目标，但第一个安全实现目标仍不清楚。立即构建很可能放大错误假设。

## 需求

先建立第一个可信证据边界，再针对这个边界做一个聚焦改动，并让 GateKeeper 按同一证据路径裁决。

## 适合这个流程，因为

Inspector 先区分事实、假设和未确认问题。Builder 再针对已落地的切片行动，而不是在整个系统里猜。GateKeeper 判断最终状态是否满足同一证据边界。

## 为什么不是其他流程

这不是 `Build + Parallel Review`，因为现在还没有足够信心先构建。也不是 `Repair Loop`，因为第二轮修复并不明确。除非稳定 benchmark 已经是主判断信号，否则也不是 `Benchmark Gate`。

## 为什么不直接交给 AI Agent

如果没有第一轮证据边界，AI Agent 很容易在错误层面上努力。Loopora 通过把证据边界外部化，减少实现前的方向漂移。

## 示例 spec

# Task

为当前问题建立第一个可信证据边界，然后实现能解决该边界的最小改动。

# Done When

- 第一个失败层或不确定层已经通过直接证据确定。
- Builder 针对该证据做一个聚焦改动，而不是广泛猜测式修改。
- 最终验证沿用塑造实现的同一证据路径。
- GateKeeper 可以说明哪些假设已经解决、哪些风险仍然存在。

# Guardrails

- 在第一证据边界清楚前，不要做大范围实现改动。
- 不要用泛泛诊断文字替代直接证据。
- 保留解释为什么选择这个切片的 artifact。

# Success Surface

- 下一步实现是被证据约束的，而不是猜出来的。
- 最终状态可以沿着稳定证据路径被评估。

# Fake Done

- 代码改了，但原来的不确定性仍然存在。
- Inspector 列出很多可能性，却没有钉住最强的第一证据边界。
- GateKeeper 因为“有改善”而通过，而不是因为已验证切片真正解决。

# Evidence Preferences

- 优先使用命令、日志、trace、测试或 artifact 对比来定位问题最先出现的位置。
- 当 workdir 没有足够证据时，必须明确标注假设。

# Role Notes

## Inspector Notes

先钉住第一个可信证据边界，再推荐实现方向。

## Builder Notes

只修复已被证据支持的切片，并让 handoff 始终回到 Inspector 证据。

## GateKeeper Notes

按同一证据边界裁决；不要让另一个成功故事替代最初证明改动合理性的证据路径。
