## 场景

当前发布分支被一个邀请链接重定向 bug 卡住了。被邀请用户会在登录页和 accept-invite 页面之间来回跳转，而不是顺利进入邀请页。范围很小，失败路径也很明确，团队要的就是一次聚焦的 Builder 热修，再加一次快速 GateKeeper 裁决。

## 需求

修好邀请链接的重定向路径，并用直接证据验证这条热修路径。

## 适合这个流程，因为

范围很小，失败路径也很明确。Builder 可以快速修掉这个重定向问题，GateKeeper 也能立刻围绕这条路径做判断，同时把热修需要的证据链留下来。

## 示例 spec

# Task

修好邀请链接重定向，让被邀请用户登录一次后就能到达 accept-invite 页面，而不是在多个路由之间循环跳转。

# Done When

- 从一个有效邀请链接出发，未登录用户只会被送去登录一次，然后到达 accept-invite 页面。
- 从同一个邀请链接出发，已登录用户会直接到达 accept-invite 页面。
- 有一条直接、可复现的验证，证明这条重定向路径不再循环跳转。
- 现有邀请 token、accept-invite 行为和相邻认证路由继续正常工作。

# Guardrails

- 改动范围只聚焦在邀请重定向和 accept-invite 路径。
- 不要把任务扩大成一轮通用认证重构。
- 保留现有邀请 token 处理、路由名称和相邻认证行为。

# Role Notes

## Builder Notes

优先选择最短、最可信的完成路径，把这次热修真正收口。

## GateKeeper Notes

仍然要有直接证据，但裁决要保持简洁，和这次热修的范围相匹配。
