## 场景

一个支持自托管的大型知识库产品，在给超大租户做全文检索重建时，经常把发布窗口拖到超时。团队已经知道，第一轮修复大概率只能先把最明显的队列堆积打掉，随后还要再看 replay、压缩或索引写入里冒出来的第二个瓶颈。

## 需求

明确留出“先修一轮、复查结果、再修一轮”的空间，让大租户的全文检索重建重新落回可接受窗口内，而不是只把问题从一个阶段挪到另一个阶段。

## 适合这个流程，因为

这类任务从一开始就知道一轮修复不太够。Builder 先打掉第一个瓶颈，Inspector 再看新的运行结果，Guide 帮忙判断第二轮应该继续压哪里，最后再由 Builder 和 GateKeeper 收口。

## 为什么不是其他流程

这不是 `Build First`，因为第一轮实现本身不是终点。它也不是 `Inspect First`，因为第一轮修复就是拿到下一轮信号的前提。更不是 `Benchmark Loop`，因为决定第二步的不是单个分数，而是第一轮修完之后剩下来的真实系统瓶颈。

## 示例 spec

# Task

把大型租户的全文检索重建重新压回可接受窗口内，并消除第一轮修复后最主要的剩余瓶颈。

# Done When

- 第一轮修复已经产出了新的重建证据，能说明最初的主要瓶颈是否被打掉。
- 复查已经确认剩余瓶颈主要落在 replay、压缩还是索引写入。
- 最终状态下，大租户重建不再拖爆发布窗口，也不会只是把阻塞从一个阶段挪到另一个阶段。
- 最新验证沿用了最初那条“大租户重建路径”，而不是换了一个更轻的替代用例。

# Guardrails

- 每一轮修复都要保持聚焦，并保留足够证据解释为什么进入下一轮。
- 保留普通租户和日常增量索引的正常行为。
- 不要为了压缩窗口而牺牲最终索引完整性或可恢复性。

# Role Notes

## Builder Notes

第一轮先打掉最主要的已知瓶颈，第二轮只处理复查后仍然存在的最大缺口。

## Inspector Notes

复查时要把新的重建轨迹和第一轮前的基线直接对比，别另起一套更轻的测法。

## Guide Notes

如果第一轮修复后冒出来的是另一类系统瓶颈，就把第二轮聚焦到那个新缺口，不要沿着旧假设继续蛮修。

## GateKeeper Notes

只有当原始发布窗口压力真正缓解，而且索引结果仍然完整可用，才允许放行。
