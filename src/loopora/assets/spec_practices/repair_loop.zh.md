## 场景

新的 hybrid search 栈在功能上已经接近可用，但在更大范围切换前还有一个明显阻塞：全量重建仍然会拖爆维护窗口。团队已经知道，第一轮修复大概率只能先打掉最明显的瓶颈，随后还会在 chunking、权限 bitmap、shard merge 或索引写入里露出第二个瓶颈。

## 需求

明确留出“先修一轮、复查结果、再修一轮”的空间，让 hybrid-search 的全量重建重新落回可接受维护窗口内，而不是只把压力从一个阶段挪到另一个阶段。

## 适合这个流程，因为

这类任务从一开始就知道一轮修复不太够。Builder 先打掉第一个瓶颈，Inspector 再看新的运行证据，Guide 帮忙判断第二轮应该继续压哪里，最后再由 Builder 和 GateKeeper 收口。

## 为什么不是其他流程

这不是 `Build First`，因为基线路径已经存在。它也不是 `Inspect First`，因为第一轮修复本身就是拿到下一轮信号的前提。更不是 `Benchmark Loop`，因为决定第二步的不是一个分数，而是第一轮修完之后剩下来的真实系统瓶颈。

## 为什么不直接交给强 Agent

强 Agent 当然可以先做第一轮修复。但之后人类还是得回来读新的 trace、判断第二个真实瓶颈是什么、再决定下一轮往哪压。这种反复回来确认和纠偏，正是 Loopora 想减少的人类流量。`Builder` 负责把新运行状态打出来，`Inspector` 负责读它，`Guide` 负责收窄第二轮方向，`GateKeeper` 负责判断窗口是不是真的被拉回来了。

## 示例 spec

# Task

把 hybrid-search 的全量重建重新压回可接受维护窗口内，并消除第一轮修复后最主要的剩余瓶颈。

# Done When

- 第一轮修复已经产出了新的重建证据，能说明最初的主要瓶颈是否被打掉。
- 复查已经确认剩余瓶颈主要落在 chunking、权限 bitmap、shard merge 还是索引写入。
- 最终状态下，全量重建不再拖爆维护窗口，也不会只是把阻塞从一个阶段挪到另一个阶段。
- 最新验证沿用了最初那条全量重建路径，而不是换了一个更轻的替代用例。

# Guardrails

- 每一轮修复都要保持聚焦，并保留足够证据解释为什么进入下一轮。
- 保留日常增量索引和线上查询服务的正常行为。
- 不要为了压缩窗口而牺牲最终索引完整性或可恢复性。

# Role Notes

## Builder Notes

第一轮先打掉最主要的已知瓶颈，第二轮只处理复查后仍然存在的最大缺口。

## Inspector Notes

复查时要把新的重建轨迹和第一轮前的基线直接对比，别另起一套更轻的测法。

## Guide Notes

如果第一轮修复后冒出来的是另一类系统瓶颈，就把第二轮聚焦到那个新缺口，不要沿着旧假设继续蛮修。

## GateKeeper Notes

只有当原始维护窗口压力真正缓解，而且索引结果仍然完整可用，才允许放行。
