## 场景

项目已经有一套客服问答评测 harness。当前基线只有 61% 通过率，而发布要求至少到 70%。团队已经说得很明确：不接受只对 benchmark 有效、却会让真实产品变差的取巧做法。

## 需求

把评估结果提升到发布线，同时保持线上检索和排序行为依然真实可信。

## 适合这个流程，因为

这类任务每一轮都该由 benchmark 结果来定方向，而不是靠直觉拍脑袋。先让 GateKeeper 看分数，再让 Builder 围绕结果继续优化；只有结果真的变了，下一轮才有必要继续。

## 为什么不是其他流程

这不是 `Build First`，因为这里驱动循环的不是“先把第一版做出来”；也不是 `Inspect First`，因为你研究的重点不是某条失败路径。它更不是 `Repair Loop`，因为每一轮最关键的问题，是分数和评测结果有没有真的变到足以改变下一步。

## 示例 spec

# Task

把项目自带的客服问答 benchmark 从当前 61% 基线提升到至少 70%，并且不引入只服务于 benchmark 的取巧做法。

# Done When

- benchmark harness 可以在仓库里端到端重跑。
- 最新评估产物是新鲜的、可检查的，并且来自当前工作区。
- 最新 benchmark 结果达到或超过 70%。
- 声称的提升能追溯到真实系统改进或 harness 可靠性改进，而不是一次性刷分技巧。

# Guardrails

- 不要硬编码 benchmark 答案、标签映射或一次性缓存。
- 在优化过程中保留现有对外产品边界。
- 保持 benchmark 输入、报告和分数产物可检查。

# Role Notes

## GateKeeper Notes

先看 benchmark 证据，不要凭直觉放行；如果 harness 自身还不可信，要先保守判定。

## Builder Notes

优先修正真实系统问题或 harness 可靠性问题，不要追逐一次性的分数尖峰。
