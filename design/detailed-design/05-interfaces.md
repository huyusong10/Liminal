# Interfaces

## 1. 模块职责

模块存在的唯一理由：

- 让不同入口以一致语义访问同一套 loop、编排与运行能力。

它负责输入规范化、边缘校验与结果投影，不负责业务编排本身。

## 2. 统一对象模型

| 对象 | 用户目的 | 稳定语义 |
|------|----------|----------|
| `loop` | 保存一套可重复执行的配置 | 绑定 workdir、spec、runtime 策略与 orchestration |
| `orchestration` | 编辑可复用 workflow | 管理角色快照、步骤顺序与收敛规则 |
| `role definition` | 预先定义可复用角色模版 | 供 orchestration 选择并复制成角色快照；同时定义默认执行配置 |
| `run` | 观察一次具体执行 | 提供状态、事件、摘要、canonical artifacts 与终端白盒输出 |
| `tutorial` | 学习正确使用顺序 | 解释角色定义 → 流程编排 → 创建循环的推荐路径，帮助用户判断某类改动应回到哪一层，并说明 context 与中间产物如何在流程中流转 |

## 3. 入口职责

| 入口 | 面向对象 | 负责内容 | 不负责内容 |
|------|----------|----------|------------|
| CLI | 终端与脚本 | 参数收集、同步/后台启动、结构化输出 | 维护独立业务状态 |
| Web UI | 可视化操作与观测 | 页面表单、列表、编辑器与实时观察 | 直接实现编排逻辑 |
| HTTP API | Web UI 与集成调用方 | 程序化创建、更新、查询与流式事件 | 暴露底层存储实现 |

## 4. 跨入口一致性

以下能力属于核心能力，必须跨入口共享同一语义：

| 能力 | 一致性要求 |
|------|------------|
| 创建 loop | 相同参数表达相同 loop definition |
| 选择 orchestration | 引用同一编排资产 |
| 配置 completion mode | `gatekeeper` 与 `rounds` 的语义一致 |
| 配置轮次间隔 | 所有入口都表达相同等待语义 |
| 编辑 orchestration | 角色快照、步骤与收敛规则语义一致 |
| 编辑 role definition | 角色模版、权限边界、执行配置与 prompt 语义一致 |
| 启动、重跑、停止、删除 run | 生命周期语义一致 |
| 校验 spec 与 prompt | 校验规则一致 |
| 暴露 spec 起草辅助能力 | 对外只使用规范名 `loopora-spec`，不再把旧品牌别名作为当前接口契约的一部分 |
| 暴露状态目录与认证入口 | 只接受 `loopora` 根目录语义与 `X-Loopora-Token` 品牌化入口，不再把旧品牌别名作为接口契约的一部分 |
| 浏览 run artifacts | 都以 canonical artifact 集合作为主视图 |
| 观察终端 | 都把系统动作、context 流转、命令和输出投影到同一事件语义 |

## 5. 允许差异

以下差异属于交互差异，允许存在：

| 范围 | 允许差异 |
|------|----------|
| Web UI | 提供可视化 workflow 编辑器、角色选择器、教程入口与实时页面导航 |
| CLI | 提供同步等待与后台执行开关 |
| Web UI | 可保存浏览器本地草稿、最近使用建议，以及用户选择的明暗主题；这些都是 best-effort |
| Web UI | 终端可以提供折叠/展开、全局展开/收缩、按类别筛选的观察控件 |
| Web UI | 角色定义页可以把预设模式与直接命令模式做成不同的视觉分区，并在界面上提供最终命令预览，只要提交语义与实际执行保持一致 |
| Web UI | 角色定义页可以自动管理 prompt 资产内部文件名，而不把它暴露成用户必须手填的字段 |
| Web UI | 角色定义页可以根据当前语言自动加载内置 prompt 的本地化版本，并在切换角色模板时同步展示对应说明与建议，只要底层角色模板语义不变；角色模板下拉标签与 archetype guide 说明块也必须跟随当前界面语言切换，不能在 locale 变更后退回固定英文；prompt 编辑区可以复用共享 Markdown workbench，在左侧编辑完整源文、右侧只预览忽略 front matter 后的正文，但提交时必须保留原始 front matter 与正文不变 |
| Web UI | 角色定义列表可以把“已保存自定义角色”和“内置角色模板”分区展示；前者进入修改流程，后者进入派生新角色流程，只要两者底层资产语义保持一致 |
| Web UI | 角色定义列表与编排列表的卡片主体可以整体作为进入编辑器的入口，但删除等危险操作必须保持为独立按钮，不能因为整卡点击而误触发或吞掉独立操作 |
| Web UI | 角色模板列表与内置编排列表可以把“打开后会派生什么”的公共说明折叠到分区标题的单个 tips 按钮中，而不是在每张卡片里重复出现，只要主操作语义仍然清晰可达 |
| Web UI | 当某个内置角色的职责容易和相邻角色混淆时，角色模板卡片可以追加该角色专属的 tips 按钮，用来解释职责边界；这类 tips 必须是角色特有信息，不能重复公共说明 |
| Web UI | 角色定义列表与编排列表可以使用响应式卡片网格：在常见桌面宽度下尽量铺满可用空间，同时保留单卡最大宽度，避免少量卡片被拉成整行长条 |
| Web UI | 编排列表可以把“已保存自定义编排”和“内置预设编排”分区展示；前者进入编辑流程，后者只读查看与派生新编排，不能在原地修改 |
| Web UI | 内置预设编排卡片可以固定显示本地化的“适用场景”说明卡；这类说明卡应表达什么时候适合选择该预设，而不是重复派生操作说明 |
| Web UI | 已保存自定义编排卡片可以省略列表页的 warning / scenario 说明卡，保留标题、描述、步骤统计、结构图与删除操作即可；这类卡片在少量项目时也应维持与内置预设编排近似的固定视觉卡宽，并保持左对齐而不是因数量变少被自动放大 |
| Web UI | 编排编辑页可以默认从空白 workflow 开始，并把载入 starter 的动作放在步骤工具条中，只要底层 workflow / preset 语义不变；starter 下拉标签应跟随当前界面语言显示，不要把中英文混排成单个选项文本 |
| Web UI | 编排编辑页可以把步骤主区做成摘要卡片，并用点击卡片切换当前角色；卡片可把 step 级会话继承、模型覆盖与附加 CLI 参数压缩成紧凑标签，并省略空值字段，只要被选中的角色快照、步骤顺序与 workflow snapshot 保持一致即可 |
| Web UI | 编排编辑页可以通过单个设置浮窗只编辑 step 级覆盖项，并把绑定的角色快照作为只读信息展示；可编辑覆盖项包括 `step.model`、step 级 session 继承开关与附加 CLI 参数，只要提交前后的 workflow / prompt 语义不变即可；这类 step 级枚举选项在首帧也应跟随当前界面语言，而不是先回落到英文再等待脚本修正 |
| Web UI | 编排页可以把 workflow 投影成循环实例图，并让节点选择与步骤卡片联动；角色快照详情可以直接显示在步骤卡片中，只要它表达的仍然是同一个 roles/steps 顺序与收敛关系；图中的节点与图例 pills 的 assistive labels 应使用当前 locale 描述“第几步 / workflow step”，同时保留内置专业角色名的英文原文 |
| Web UI | 创建循环页可以把说明文案收进字段就近的 tips 按钮，并按所选编排动态隐藏不适用的运行策略字段；Spec 字段旁可以提供共享 Markdown workbench 形态的弹窗编辑器，用同一路径读取、编辑并保存 `spec.md`，同时展示校验状态，并允许用户在“源码编辑”和“手动打开的渲染预览”之间切换；弹窗里的保存行为必须明确写回磁盘，提交 loop 时读取的仍是同一路径下的 spec 文件，不能制造“弹窗内容”和“实际提交内容”分叉；编辑器头部说明与操作区不应互相挤压，长 spec 必须在弹窗正文内部滚动而不是把整页或整窗撑乱；运行策略枚举与页面标题在首帧也应跟随当前界面语言，而不是先回落到英文再等待脚本本地化 |
| Web UI | 创建循环页与编排编辑页的工作台头部可以把说明文案与跨页捷径放在同一个 header shell 里；这些捷径只能作为次级入口，不能盖过当前表单的主任务，也不能改变当前 loop / workflow 的提交语义 |
| Web UI | 教程页可以使用独立的阅读壳层，并在大屏上适度放宽 guide/context 面板的整体宽度，只要“三层职责分流 + context 流转”这套信息顺序与锚点结构保持不变 |
| Web UI | 可以为一级导航、卡片和可视化视图增加轻量动效，只要不改变页面语义或主操作路径；一级导航 hover / active 应保持单层强调面，避免视觉叠层 |
| Web UI | 主题与语言偏好可以跨页面持久化，但必须在首个可见帧前完成初始化；所有页面壳层（包括认证页与 fullscreen console）都必须共用同一套首屏 bootstrap 和请求 locale 归一化规则，首选请求 locale 需要按 `Accept-Language` 权重解析，不能因为 header 里较低优先级的 fallback 语言先出现就被反向覆盖；共享顶栏、显示偏好控件、控制台工具条以及帮助提示按钮这类首屏可达控件的 assistive labels 也必须同步跟随当前 locale，而不是固定英文；纯装饰性的品牌图不应再额外播报固定语言的 logo 文案；不能依赖整页先隐藏再淡入的挂载动画来修正首屏状态，且深色主题下的卡片说明、流程图标签和状态文案必须保持足够文字对比度 |
| Web UI | 小屏与中屏可以把主题、语言等全局偏好折叠到顶栏次级菜单中；一级功能导航可改成多行网格，只要它仍保持优先可达、不与偏好控件重叠，并保持足够文字对比度 |
| Web UI | 顶层对象页与教程/工具等一级入口页应保留明确的大标题，并可配合说明文案一起表达当前对象、主操作与推荐路径；对应文档标题也应在首个可见帧就跟随当前 locale，不能固定为英文 |
| Web UI | 已创建循环列表可以使用固定视觉卡宽、左对齐的卡片轨道；运行中状态的视觉强调必须被限制在卡片内部，不能扩散到整页背景，并且应优先使用稳定的表面/边框强调而不是周期性扫光、脉冲等会制造闪烁感的动画；卡片主体可以直接进入最新 run 或 loop 详情，但查看/删除等独立操作必须继续保留独立点击目标 |
| Web UI | loop 详情页可以把 loop-level 信息拆成配置、原始 spec、最近摘要和运行历史等独立面板；原始 spec 面板可以复用共享 Markdown workbench 的只读形态，在左侧保留原文、右侧展示渲染结果；原文、渲染结果与长路径都必须在当前卡片内保持可读和可滚动，但不能改变 loop definition 的语义或来源 |
| Web UI | run 详情页不应再把 `.loopora` 原始文件直接当成主信息层浏览；“关键产物”必须收敛成按 iteration 聚合的角色结论面板，并通过轮次下拉在单轮视图间切换，默认落在最新一轮，只展示该轮各角色 handoff / GateKeeper verdict / score 这类用户决策真正需要的摘要，原始 prompt、context、output 继续留在 `.loopora` 日志目录；页面可以提供直达构建目录与 `.loopora` 日志目录的快捷入口，但不能让长 JSON 预览重新成为主视图；当单条角色结论过长时，应使用卡片内滚动而不是省略号截断整段结论 |
| Web UI | run 详情页的控制台与时间线可以采用高密度横向布局：控制台行按时间/标签/内容对齐，筛选 chips 可以直接承担颜色与开关语义；控制台筛选应优先贴近用户心智模型，默认折叠为 `Status / Actions / Result` 这类少量分组，而不是把底层事件类型逐个暴露出来；fullscreen console 可以收掉标题、摘要与统计卡，只保留返回入口，并让终端区域接管整页可滚动空间 |
| Web UI | run 详情页的运行进度可以展示为分层阶段图：`Checks` 与 `Done` 作为 terminal nodes，中间 workflow steps 作为独立 loop lane；实现可以通过更紧凑的 station / rail 视觉比例、terminal 对位和 loopback connector / arc 装饰来优化美观度，但阶段数量、顺序与 loop 关系都必须按当前 run 的 workflow snapshot 渲染，而不是按 archetype 写死；如果当前 snapshot 没有中间 workflow steps，中央 loop lane 应退化成说明型 empty state，并去掉会误导用户以为还有中间步骤的 connector / rail / arc 装饰；这个 empty state 的标题与说明也必须在首个可见帧就跟随当前页面 locale，而不是先落成英文再等待前端脚本修正；进度头部不应重复展示已经在 live card 或 stage chip 中可读的耗时摘要；深色主题下 terminal / workflow / highlight 卡片也必须保持同一套暗色表面语言，不能退回浅色反白块 |
| Web UI | 内置专业角色名在所有 locale 下都保持英文原文（如 `Builder`、`Inspector`、`GateKeeper`、`Guide`）；只有用户自定义角色名可以按用户输入原样展示 |
| 不同入口 | 可使用不同文案、布局与交互节奏，只要不改变底层语义 |

## 6. 不允许差异

以下差异不可接受：

- 只有某一入口能表达核心 loop 配置
- 只有某一入口能创建或编辑 orchestration / role definition
- 同名配置项在不同入口含义不同
- 预览行为与实际提交行为不一致
- 在 orchestration 中直接修改 role definition 才拥有的字段
- 在 loop 创建中重新定义角色默认执行配置
- 只有某一入口能表达 step 级 session 继承或附加 CLI 参数
- 把内部存储标识直接当作主要用户文案
- 明暗主题切换改变了页面语义或交互路径
- 没有语义理由的一级页面主内容宽度不一致

## 7. 安全边界

| 边界 | 接口层责任 |
|------|------------|
| 本地与网络访问 | 网络暴露必须显式开启，本地默认安全优先；网络模式下的认证引导页可以按请求 locale 与本地显示偏好初始化，但查询参数、header 与 cookie 的鉴权语义必须保持一致 |
| 文件系统访问 | 所有路径都必须受允许根范围约束 |
| 文件预览 | 按不可信内容处理，不能直接执行 |
| 输入规范化 | 非法、缺失或越界输入在接口层被拦截或标准化 |

## 8. 依赖方向

依赖方向必须保持为：

`用户交互 → 接口层 → orchestration service`

禁止：

- Web 直接实现业务编排
- CLI 维护独立状态模型
- API 直接暴露持久化层细节

## 9. 变更触发

以下变化需要更新本文档：

- 新增或删除顶层入口
- 跨入口能力模型变化
- 某个核心对象只能在单一入口创建或编辑
- 角色定义 / 编排 / 创建循环的责任边界变化
- 安全边界变化

以下变化通常不需要更新本文档：

- 页面改版
- 帮助文案调整
- 局部字段扩展

## 10. 非目标

- 不把 Web 做成通用 IDE
- 不让 CLI 承担复杂可视化职责
- 不把 HTTP API 扩张成公网平台接口
