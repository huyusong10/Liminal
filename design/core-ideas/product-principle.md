# Product Principle

## 1. 最高产品原则

Loopora 是面向长期 AI Agent 任务的本地工作台：

> 把任务级判断编译成可读、可运行、可取证、可修订的循环系统。

更短地说：

> Loopora is a task-scoped harness compiler + evidence loop runtime.

所有产品、接口、运行时和文档设计都必须服务于这条主线：

`任务输入 -> 对齐成循环方案 -> 运行并收集证据 -> 基于反馈修订方案`

如果某个能力不能帮助用户更好地完成这条闭环，它就不应成为默认主路径。

## 2. 独特性

Loopora 不只是把多个 AI Agent 角色排在一起。

传统多角色 harness 往往强调角色库、命令库、技能注入或专家对抗；它们能让 Agent 更像一个工程团队，但通常不把“这次任务如何判断完成”编译成一个可管理、可预览、可修订的对象。

Loopora 也不只是循环执行器。

纯循环工具抓住了“执行 -> 检查 -> 再执行”的节奏，但通常缺少任务契约、角色姿态、workflow 时序、证据账本和方案 revision 之间的整包关系。

Loopora 的差异来自五件事必须同时成立：

1. 用户先描述长期任务，而不是先手工拼角色或流程。
2. alignment 先形成本次任务的判断方式，再生成循环方案。
3. 循环方案通过 `spec / roles / workflow` 三种运行面表达同一份姿态。
4. run 不能只留下日志，必须留下能回答“为什么过 / 为什么没过”的证据。
5. 用户反馈应进入下一版方案，而不是让用户从零重写 prompt 或手改零散字段。

## 3. 默认用户心智

新用户不应该被要求先理解 `bundle`、`spec`、`roles`、`workflow` 或 provider 参数。

默认心智应是：

1. 我有一个长期任务。
2. Loopora 帮我问清楚什么算真实进展。
3. Loopora 生成一份循环方案。
4. 我预览方案，确认它如何执行、检查和裁决。
5. Loopora 运行方案并留下证据。
6. 我根据证据反馈，得到下一版方案。

`bundle` 是内部交换单元和专家资产；面向新手的主语言应优先使用“循环方案 / 任务方案 / 方案库 / 证据 / 修订”。

## 4. 稳定运行面

Loopora 不执行一个抽象的“姿态”对象。

每次任务姿态必须被编译到以下运行面：

| 运行面 | 职责 |
|--------|------|
| `spec` | 冻结任务契约、成功面、假完成、证据偏好、残余风险 |
| `roles` | 冻结各 AI Agent 角色在本任务中的工作姿态 |
| `workflow` | 冻结判断发生的顺序、纠偏入口和收束方式 |
| `evidence` | 记录每轮行动、检查、裁决和残余风险 |
| `revision` | 把用户反馈转成下一版方案，而不是零散修补 |

前三者是运行输入；后两者是长期任务闭环的复盘与演进机制。

## 5. 反目标

Loopora 不应退化成：

- role zoo：堆很多角色，但任务判断不可解释。
- prompt pack：只提供更长提示词，但没有运行面和证据闭环。
- loop script：重复执行命令，但缺少 task contract、workflow posture 和 revision。
- chat wrapper：提供通用聊天体验，但绕过方案预览、bundle 生命周期和 run evidence。
- asset CRUD console：暴露大量内部对象，让用户忘记自己是在管理长期任务。

## 6. 设计判断式

做产品和实现决策时，优先问：

1. 这是否让用户更清楚“什么算真实进展”？
2. 这是否让 AI Agent 的行动、检查、裁决分工更可复盘？
3. 这是否产生新的可用证据，而不是只产生更长日志？
4. 这是否支持从反馈进入下一版方案？
5. 这是否把内部概念留给专家路径，而不是压到新手主路径？

若答案是否定的，它不应进入默认体验。
