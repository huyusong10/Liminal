# Product Principle

## 1. 最高产品原则

Loopora 是面向长期 AI Agent 任务的外部任务治理层。

它的核心不是“让 Agent 更会写 prompt”，而是：

> 把任务治理从 Agent 的上下文里拿出来，变成外部、持久、可检查、可运行、可修订的控制系统。

英文压缩定义：

> Loopora is an external task-governance harness for long-running AI Agent work.

所有产品、接口、运行时和文档设计都必须服务于这条闭环：

`任务输入 -> 对齐治理方案 -> 运行并收集证据 -> 基于证据修订 harness`

## 1.1 五分钟上手原则

Loopora 可以越来越强，但默认第一次使用必须仍然像一句话启动长期任务。

稳定原则：

- 新用户只需要描述任务、选择 workdir、确认循环方案、运行、看证据、修订。
- 高级能力必须由 Loopora 编译进循环方案，而不是要求用户手动配置一套自动化系统。
- 任何新增能力进入默认主路径前，都必须证明它服务于误差控制，并且不增加首次使用心智负担。
- `workflow controls`、触发器、计时器、并行检视和信息流策略属于专家预览、方案详情或修订路径；它们不应成为新手必学概念。
- README、教程和 Web 首屏必须继续围绕“任务 / 循环方案 / 证据 / 修订”，而不是把内部字段当作入口词。

## 2. 为什么这不是普通 Agent 插件

传统 Agent 插件通常在 Agent 内部增强能力：加入 skill、命令、角色、检查表或优秀协作模式。它们让 Agent 更有纪律，但控制权仍主要留在 Agent 上下文里。

Loopora 的差异在于把控制结构外部化：

| 风险 | 只靠 Agent 上下文 | Loopora 的外部治理 |
|------|-------------------|--------------------|
| 目标漂移 | 依赖模型记住原意 | `spec` 冻结任务契约和边界 |
| 假完成 | 依赖 prompt 要求自查 | `GateKeeper` 与证据门槛阻断收尾 |
| 判断混杂 | 一个 Agent 同时写、查、判 | `roles` 分离构建、检查、裁决和纠偏 |
| 循环空转 | 重复调用但新证据不足 | `workflow` 规定何时取证、何时停止 |
| 经验丢失 | 反馈变成下一段聊天 | `revision` 修改 harness 本身 |

因此，Loopora 不承诺所有任务都比插件更适合。它只适合那些需要把判断、证据和停止条件从一次对话里外化出来的长期任务。

## 3. 核心对象

Loopora 的用户心智是“循环方案”，内部交换单元是 `bundle`。

一份循环方案必须同时表达五个运行面：

| 运行面 | 职责 |
|--------|------|
| `spec` | 冻结任务范围、成功面、假完成、guardrails、证据偏好与残余风险 |
| `roles` | 冻结各 AI Agent 角色在本任务中的构建、检查、裁决或纠偏姿态 |
| `workflow` | 冻结判断顺序、handoff、纠偏入口和收束方式 |
| `evidence` | 记录每轮行动、检查、裁决、失败原因和未证明内容 |
| `revision` | 把用户反馈和 run evidence 转成下一版 harness |

前三者是运行输入；后两者是长期任务继续变好的条件。

## 4. 默认用户心智

新用户不应该被要求先理解 `bundle`、`spec`、`roles`、`workflow` 或 provider 参数。

默认心智应是：

1. 我有一个长期任务。
2. Loopora 帮我问清楚什么算真实进展、什么是假完成、信任什么证据。
3. Loopora 生成一份循环方案。
4. 我预览它如何执行、检查、裁决和停止。
5. Loopora 运行方案并留下证据。
6. 我根据证据反馈，修订下一版方案。

`bundle` 是专家资产和内部交换单元；默认界面应优先使用“任务 / 循环方案 / 方案库 / 证据 / 修订”。

## 5. 反目标

Loopora 不应退化成：

- role zoo：堆角色，但任务判断不可解释。
- prompt pack：写更长提示词，但没有外部运行面和证据闭环。
- loop script：重复执行命令，但缺少 task contract、GateKeeper 和 revision。
- chat wrapper：提供通用聊天体验，但绕过方案预览、生命周期和 run evidence。
- asset CRUD console：暴露大量内部对象，让用户忘记自己是在治理长期任务。

## 6. 设计判断式

任何产品和实现决策都应优先问：

1. 这是否把关键判断从 Agent 上下文里外部化了？
2. 这是否让“什么算真实进展 / 假完成 / 可接受风险”更可检查？
3. 这是否产生新证据，而不只是更长日志？
4. 这是否让用户反馈能修订 harness，而不是只追加 prompt？
5. 这是否避免把内部概念压到新手主路径？

若答案是否定的，它不应进入默认体验。

新增高级能力还必须额外回答：

6. 它是否仍然保持五分钟上手？
7. 它是否只是帮助控制长期任务误差，而不是把 Loopora 推向通用自动化平台？
