# 核心思想

## 0. 最高产品原则

Loopora 是面向长期 AI Agent 任务的外部任务治理层。

核心闭环是：

`任务输入 -> 对齐治理方案 -> 运行并收集证据 -> 基于证据修订 harness`

`product-principle.md` 是本目录的最高原则。下方公理都服务于同一件事：让长期任务的控制结构不只依赖 Agent 自律。

如果讨论卡在 `bundle / loop / run / harness / alignment session` 等术语层级上，先读 `concept-map.md`。它只负责澄清概念分层和主生命周期，不新增运行契约。

## 1. 设计公理

1. **治理必须外部化。**
   关键判断不能只藏在当前 Agent 对话里，必须能被保存、预览、运行和修订。

2. **loop 的理由是新证据。**
   如果下一轮不会产生新的外部反馈，loop 就会从收敛器退化成随机游走。

3. **生成器不能裁决自己。**
   Builder 可以改动；Inspector 负责取证；GateKeeper 负责是否结束。

4. **任务契约必须冻结。**
   `Task`、checks、guardrails 和完成边界在一次 run 内不能被角色静默改写。

5. **判断必须落到运行面。**
   用户姿态必须投影到 `spec / roles / workflow`，不能只留在 prompt 文案。

6. **证据必须可追溯。**
   每次 run 必须留下结构化产物、事件流和摘要；没有产物就等于没有发生。

7. **反馈应修订 harness。**
   run 后反馈不应只是“再提示 Agent 一次”，而应能改变下一版循环方案。

8. **workdir 是用户资产。**
   Loopora 可以修改目标目录，但不得把破坏性重建当作默认策略。

9. **接口层必须同构。**
   Web、CLI、API 表达同一套 loop 和 bundle 语义，不允许核心能力长期只存在于单一入口。

10. **系统首先是本地治理器。**
   状态、锁、日志、证据和产物优先围绕本地目录和本地数据库组织。

## 2. 不变量

- 单个 workdir 同一时刻最多允许一个活动 run。
- run 的主要事实源必须可从 `.loopora/` 与 `app.db` 还原。
- `Inspector` / `GateKeeper` / `Guide` 默认不应修改源文件。
- `GateKeeper` 的通过结论必须可映射到 checks、metrics、证据或边界约束。
- `Role Notes` / `posture_notes` 只能改变角色工作姿态，不能静默改变全局成功标准。
- `.loopora/` 是系统保留空间；业务改动不应写入该目录。

## 3. 反例

| 反例 | 违反规则 | 后果 |
| --- | --- | --- |
| 让 Agent 在没有新证据的前提下继续空转几轮 | loop 的理由是新证据 | 叙事漂移，任务变成盲盒 |
| 每轮都重新生成 checks | 任务契约必须冻结 | 分数不可比较，停滞检测失真 |
| Builder 自己宣布完成 | 生成器不能裁决自己 | 通过条件退化成自我报告 |
| Inspector 为了通过检查而改代码 | 角色职责混杂 | 证据与修复混在一起 |
| 只在 prompt 里描述用户偏好 | 判断没有运行面 | 长任务中容易遗忘或漂移 |
| 只记录日志，不留 evidence artifact | 证据不可追溯 | 无法复盘、比较、修订 |
| Web 能做核心能力，CLI/API 无语义入口 | 接口层不同构 | 系统出现双重能力模型 |

## 4. 非目标

- 不是通用 DAG 编排引擎。
- 不是多租户远程 SaaS。
- 不是长期人格记忆系统。
- 不是 benchmark 刷分器。
- 不是替代底层 AI Agent CLI 的二次聊天界面。

## 5. 压缩判断式

- 如果某条信息不能帮助“外部化治理 / 冻结契约 / 复盘证据 / 修订 harness”，它就不属于核心设计。
- 如果某个能力只能在一个入口存在，它就不是完成态。
- 如果一个 run 结束后不能回答“改了什么、怎么测的、为什么过/没过、下一版 harness 要不要改”，该设计就是不合格的。
