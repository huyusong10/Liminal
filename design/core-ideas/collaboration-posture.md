# Collaboration Posture

## 1. 目标

本文档定义 Loopora 在下一阶段的核心产品方向：

- Loopora 不只是在编排 workflow。
- Loopora 更重要的职责，是帮助用户外化自己的协作人格、判断习惯与工作偏好。
- `workflow`、`prompt`、`role` 应逐步从“用户手工拼装的起点”退居为“由 posture 编译出来的结果”。

一句话概括：

> Loopora 应从 workflow tool 演进为 collaboration posture compiler。

## 2. 问题判断

用户之所以想使用 Loopora，通常不是因为他已经清晰掌握一套固定规则，而是因为：

- 他已经和 AI 形成了一种稳定但难以完整言明的协作定式。
- 他隐约知道自己会在什么时刻不信任结果，什么时刻愿意继续推进。
- 他能感受到“这次太草率了”“这次不够重视重构”“这次查得很全但太保守”，却未必能把这些感觉直接翻译成规则系统。

因此，Loopora 的关键任务不是要求用户先写出完整流程规则，而是帮助用户回答更接近真实协作的问题：

- 我默认会怀疑什么？
- 我最信任哪类证据？
- 我如何区分“小瑕疵”和“方向错了”？
- 我偏好保守签字、快速试探，还是先做完整清理？
- 我最讨厌哪类假进展？

这些信息比 workflow rule 更接近用户真正想自动化的部分。

## 3. 核心判断

### 3.1 一等资产应当变化

当前系统的一等资产主要是：

- `orchestration`
- `workflow`
- `role definition`
- `prompt`

下一阶段，Loopora 还需要把以下内容提升为一等资产：

- `posture profile`
- `posture delta`
- `posture cases`

其中：

- `posture profile` 用来表达用户稳定的协作姿态。
- `posture delta` 用来表达用户对某次 run 的模糊但高价值纠偏。
- `posture cases` 用来表达“像我会这么判断”的具体判例，而不是抽象自述。

### 3.2 用户真正想自动化的是判断习惯

Loopora 主要抽取的对象不应是 workflow rule，而应是 collaboration posture。

这里的 posture 指：

- 工作气质
- 风险偏好
- 证据偏好
- 判断阈值
- 角色间的协作姿态
- 对“完成”“敷衍”“过度设计”“偷懒”“缺乏重构”“证据不足”等状态的主观敏感度

它描述的不是“第几步做什么”，而是“这个用户会如何理解和判断第几步的结果”。

### 3.3 workflow 是衍生物，不是入口

理想入口应当是：

`帮助用户认识自己 -> 生成 posture -> 编译 workflow / prompts -> 用户查看并修改`

而不是：

`先选择 workflow -> 再理解角色 -> 再手写 prompt`

workflow 依然重要，但它更适合作为 posture 的结构化投影，而不是用户首先被要求掌握的工具语言。

## 4. Posture 的表达原则

### 4.1 不以规则为唯一入口

规则驱动仍然是有效方法，但不应成为唯一方式。

因为用户更容易表达的是：

- 性格
- 气质
- 偏好
- 判例
- 不满意的感觉

而不是完整、稳定、可枚举的规则集。

### 4.2 优先表达“人格 + 判例”，再落到规则

纯规则容易僵硬。
纯人格容易漂移。

更稳定的表达应由四部分组成：

- 人格或气质：这个角色像怎样的同事或怎样的工作风格。
- 判断偏好：默认怀疑点、最信任的证据、对残余风险的容忍方式。
- 典型判例：哪些结果像“会放行”，哪些结果像“会卡住”。
- 少量硬边界：明确不能突破的安全边界、任务边界和外部限制。

Loopora 不应要求用户直接写出大而全的 policy，而应逐步帮助用户把这些信息沉淀成可复用资产。

### 4.3 案例比抽象自述更可信

很多姿态信息难以从抽象自述稳定获得，但可以从案例中提取。

例如：

- 哪个 verdict 更像用户会说的话。
- 面对残余风险时，用户更可能接受哪种解释。
- 面对“能跑但很脆”的结果时，用户会偏向先上线、先补证据，还是先重构。

因此，判例不是 prompt 的装饰，而应成为 posture 资产的重要来源。

## 5. Posture 与 Prompt / Workflow 的关系

### 5.1 prompt 不是 posture 本体

prompt 只是 posture 注入执行系统的一种方式，不是 posture 本身。

posture 至少应影响：

- workflow 的生成
- role prompt 的编译
- 证据优先级
- verdict 的表达方式
- 继续推进 / 暂停 / 转向的时机
- 系统给用户的解释层

如果 posture 只停留在 prompt 文案，系统仍会过度依赖手工调 prompt，难以沉淀为稳定产品能力。

### 5.2 不同角色会呈现同一 posture 的不同侧面

同一个用户的 posture，不会在所有角色上以同样方式出现。

例如：

- 在 `Inspector` 上，可能体现为“证据洁癖”“先找最大风险”“更信重复性证明而不是演示”。
- 在 `GateKeeper` 上，可能体现为“签字保守度”“对残余风险的容忍度”“对重构质量是否敏感”。
- 在 `Guide` 上，可能体现为“遇到停滞时偏向换方向还是缩小切口”。

因此，系统不应只保存一段总 prompt，而应允许 posture 被按角色编译成不同工作姿态。

## 6. Posture 与 Contract 的关系

### 6.1 Contract 提供稳定性，Posture 提供涌现

`contract` 的价值是稳定输出边界。
`posture` 的价值是引入更接近真实人的判断差异与涌现。

Loopora 不应追求把所有结果都压成完全一致。人类同一件事干两次，结果本来也可能不同。

系统应允许 posture 带来以下变化：

- 对风险排序的变化
- 对证据强弱的判断变化
- 对重构、清理、验证深度的偏好变化
- 对继续推进或暂停的倾向变化

### 6.2 变化必须可理解、可追溯

系统允许变化，不等于允许静默漂移。

看不见来源的变化是随机；
看得见来源的变化才是人格。

因此，当 posture 影响一次运行结果时，系统应尽量回答：

- 这次为什么比上次更强调重构？
- 这次为什么更保守或更激进？
- 这次为什么没有直接放行？
- 这次的 workflow 和 verdict 是由哪类 posture 信号影响的？

### 6.3 当 posture 触碰 contract 时，应显式暴露

Loopora 可以允许 posture 去挑战 contract，甚至提出“当前冻结的目标或标准可能不合理”。

但如果 posture 想改变以下内容，就不应静默发生：

- `Task`
- `Done When`
- 明确安全边界
- 用户已冻结的关键约束

这类变化应被显式提出来，作为待确认的偏移，而不是隐藏在一次普通 run 中。

## 7. Loopora 应如何帮助用户认识自己

### 7.1 目标不是让用户填更多表单

系统不应把“认识自己”退化为一组复杂配置项。

更合理的方式是帮助用户完成外化，而不是要求用户先完成抽象。

### 7.2 更自然的外化材料

Loopora 更应从以下材料中理解用户：

- 用户对案例的选择
- 用户对系统 verdict 的修改
- 用户对某次 run 的不满反馈
- 用户的模糊语义纠偏
- 用户对“像我 / 不像我”的判断

例如：

- “这次不够重视重构。”
- “这次太偷懒了，完全没有把活细致地干好。”
- “这次判断太保守了。”
- “我其实愿意接受这个 residual risk。”

这些不是低质量反馈，而是极高价值的 posture 信号。

### 7.3 最真实的姿态来自 run 后纠偏

用户在 onboarding 时对自己的理解，往往是概括性的。

用户在看完一次实际 run 后给出的反馈，往往更真实：

- 他会对哪种问题敏感。
- 他会如何评价一次“表面正确但不够满意”的结果。
- 他会如何区分“完成了”和“只是糊弄过去了”。

因此，Loopora 不应把 posture 生成视为一次性动作，而应把它视为持续学习过程。

## 8. 持续学习不是“重新生成”，而是“微调”

许多编排系统的问题在于：

- 生成一次
- 运行一次
- 用户不满意
- 只能整体重来

Loopora 应避免这种“一锤子买卖”。

更合理的方向是：

- 允许用户用模糊语义表达不满
- 把这种不满转化成 `posture delta`
- 在后续 run 中局部吸收，而不是整体推倒重来

这类微调能力应成为系统核心能力，而不是附加功能。

## 9. 产品判断

如果 Loopora 继续把主要价值放在 workflow 编排本身，它会是一套强但偏冷的 expert tool。

如果 Loopora 能做到以下事情，它才会形成真正差异化：

- 帮助用户认识自己的协作姿态
- 帮助用户把模糊的人格、气质与偏好转成可复用资产
- 自动把 posture 编译成 workflow、prompt 与 verdict 风格
- 在每次 run 后持续吸收用户纠偏
- 让用户看见“系统为什么越来越像自己”

届时，Loopora 的核心价值就不只是“自动化 agent workflow”，而是：

> 帮助用户外化自己的 collaboration posture，并把它持续注入 AI 执行系统。

## 10. 非目标

- 不是把所有用户都压成同一种最佳实践。
- 不是把人格化退化成更长的 prompt 文案。
- 不是用模糊姿态替代所有 contract。
- 不是为了“更像人”而放弃可复盘、可解释、可追溯。
- 不是要求用户在第一次使用时就完整认识自己。
